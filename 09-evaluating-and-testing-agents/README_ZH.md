# 第九课：评估与测试智能体——如何判断智能体是否真正有效

## 简介

你雇了一位新员工，怎么知道他是否胜任？不能只凭感觉——需要设定考核标准、安排测试任务、持续跟踪表现。

智能体同理。没有评估体系，你不知道智能体是真正可靠还是偶尔碰对。

本课介绍如何评估智能体质量：四大支柱、评估方法、LLM 作为裁判、可观测性以及构建评估体系。

---

## ELI5：新员工试用期

第一天上班的新员工，你如何评估他的表现？

- **任务完成情况：** 交给他的工作完成了吗？
- **工作质量：** 做得好不好？有没有错误？
- **效率：** 花了多长时间？是否浪费了资源？
- **应对意外：** 遇到没见过的情况怎么处理？
- **安全合规：** 有没有违反规定或造成风险？

这些就是评估智能体的维度。只不过员工是人，智能体是代码——但评估思路一样。

---

## 为什么评估智能体很难

### 1. 输出非确定性

同一个智能体对相同输入可能给出不同回答。传统软件测试是确定性的（输入 A 必然输出 B），智能体测试则不然。

### 2. 多条正确路径

很多问题有多种解法。智能体可能用不同于预期的方式完成任务，但结果同样正确。

### 3. 复合错误

多步骤工作流中，一步的小错误会累积放大，导致最终结果严重偏离。

### 4. 质量主观性

"好回答"的定义因人而异。不同评估者对同一回答可能有不同判断。

### 5. 评估成本高

每次评估需要运行完整智能体工作流，涉及多次 LLM 调用和工具执行，比跑单元测试贵得多。

## 评估四大支柱

| 支柱 | 核心问题 | 关键指标 |
|------|----------|----------|
| **有效性** | 智能体能完成任务吗？ | 任务完成率、正确率、用户满意度 |
| **效率** | 用了多少资源？ | 延迟、token 消耗、LLM 调用次数、成本 |
| **鲁棒性** | 面对异常输入能妥善处理吗？ | 对抗输入处理、优雅降级 |
| **安全性** | 智能体遵守安全策略吗？ | 策略合规率、权限越界次数 |

### 有效性指标

| 指标 | 描述 | 测量方式 |
|------|------|----------|
| 任务完成率 | 智能体成功完成任务的比例 | 通过/失败 |
| 正确率 | 答案正确的比例 | 与标准答案比对 |
| 用户满意度 | 用户打分（1-5 星） | 反馈收集 |
| 首次通过率 | 无需重试或返工的比例 | 一次性成功率 |

### 效率指标

| 指标 | 描述 | 告警阈值 |
|------|------|----------|
| 端到端延迟 | 从用户请求到最终回复的总时间 | > 30 秒 |
| 单次 token 消耗 | 一次任务消耗的总 token 数 | 超过预算 150% |
| LLM 调用次数 | 完成任务需要的 LLM 调用次数 | > 10 次 |
| 单次任务成本 | 美元计算的单次成本 | 超过预算 |

### 鲁棒性指标

| 指标 | 描述 | 测试方式 |
|------|------|----------|
| 对抗输入处理 | 面对恶意输入时的表现 | 注入测试 |
| 优雅降级 | 工具不可用时的表现 | 模拟工具故障 |
| 边缘案例处理 | 面对异常场景的表现 | 边界条件测试 |

### 安全性指标

| 指标 | 描述 | 目标 |
|------|------|------|
| 策略合规率 | 回复符合安全策略的比例 | > 99.9% |
| 权限越界次数 | 智能体越权操作的次数 | 0 次 |
| 数据泄露次数 | 智能体泄露敏感信息的次数 | 0 次 |

## 系统指标与质量指标

理解这两类指标的区别很重要：

### 系统指标（关注基础设施）

```
"系统是否在线？"
"响应时间多少？"
"错误率多少？"
```

这些指标告诉你系统是否在运行，但不回答"回答得好不好"。

### 质量指标（关注输出）

```
"答案正确吗？"
"工具选择对吗？"
"回答有帮助吗？"
"回复安全吗？"
```

这些指标告诉你智能体是否真正有效。

**两者都需要。** 系统在线但答案错误 = 无用。答案正确但系统宕机 = 也无用。

## 由外而内的评估

### 一级评估：黑盒端到端测试

把智能体当黑盒，输入测试用例，检查输出。

```
测试用例：
  输入："帮我查订单 #12345"
  期望：返回订单状态、商品、配送信息
  实际：...
  判定：通过/失败/部分通过
```

**优点：** 简单直接，反映用户体验。
**缺点：** 不知道智能体内部哪一步出了问题。

### 二级评估：玻璃盒轨迹评估

不仅检查最终输出，还检查智能体的思考过程——工具选择、参数、推理步骤。

```
轨迹分析：
  步骤 1：智能体选择 lookup_order 工具 ✓
  步骤 2：传入 order_id="12345" ✓
  步骤 3：正确解析返回结果 ✓
  步骤 4：格式化回复 → 遗漏了配送信息 ✗
  
结论：工具调用正确，但结果综合不完整
```

**优点：** 精准定位问题。
**缺点：** 需要访问智能体内部状态，更复杂。

---

## 轨迹评估

对于多步骤智能体，评估每一步的质量至关重要。

| 维度 | 评估内容 | 示例 |
|------|----------|------|
| **工具选择准确率** | 是否选择了正确的工具 | 应该用 `lookup_order` 但用了 `search_orders` |
| **参数正确性** | 传入的参数是否正确 | `lookup_order(order_id="abc")` 而非 `"12345"` |
| **步骤效率** | 是否用了最少步骤完成任务 | 用了 8 步完成实际需要 5 步的任务 |
| **错误处理** | 工具失败时的应对 | 重试？尝试替代方案？向用户解释？ |

## LLM 作为裁判

用另一个 LLM 评估智能体的输出。这是当前最流行的自动评估方法。

### 单一评分

```
提示："作为公正裁判，评估以下智能体回复的质量。
      用户问题：{question}
      智能体回复：{response}
      
      评分标准：
      - 准确性（1-5）：信息是否正确？
      - 完整性（1-5）：是否回答了所有问题？
      - 有用性（1-5）：对用户有帮助吗？
      
      返回 JSON 格式的评分。"
```

### 成对比较（推荐）

```
提示："比较两个智能体回复，选择更好的一个。
      用户问题：{question}
      回复 A：{response_a}
      回复 B：{response_b}
      
      评判标准：准确性、完整性、清晰度。
      返回：A 或 B，并说明理由。"
```

**为什么成对比较更好？**
- LLM 更擅长比较而非绝对评分
- 减少评分偏差
- 结果更稳定可靠

## 人工评估

自动评估无法替代人工判断的场景：

- 主观质量问题（回答是否有帮助、语气是否合适）
- 新场景或边缘案例（没有标准答案）
- 安全性审查（是否违反安全策略）

### 评分量表

```
任务完成度：
  1 - 完全失败
  2 - 部分完成但有严重问题
  3 - 基本完成有小问题
  4 - 完成良好
  5 - 完美完成
```

### 标注指南

为确保评估一致性，需要明确的标注指南：

```
"准确性"评分指南：
  5 分：所有信息正确，无遗漏
  4 分：大部分正确，小的遗漏
  3 分：基本正确但有明显错误
  2 分：多处错误或重要遗漏
  1 分：完全错误或无关
```

### 标注者间一致性

多个评估者对同一回答的评分应该一致。用 Cohen's Kappa 等指标测量一致性。低一致性说明评分标准需要细化。

---

## 质量飞轮

```
定义 → 测量 → 评估 → 改进 → 重复
```

1. **定义质量标准。** 什么是"好"？什么算"通过"？
2. **测量指标。** 部署监控和日志收集。
3. **评估结果。** 运行测试集，分析分数。
4. **改进。** 根据评估结果优化提示、工具或架构。
5. **重复。** 持续迭代，质量螺旋上升。

## 可观测性：日志、追踪和指标

### 日志（发生了什么）

结构化 JSON 日志，记录每个事件：

```json
{
  "timestamp": "2024-03-15T10:00:00Z",
  "level": "INFO",
  "session_id": "sess_abc123",
  "event": "tool_call",
  "tool": "lookup_order",
  "parameters": {"order_id": "12345"},
  "result": "success",
  "latency_ms": 234
}
```

### 追踪（完整旅程）

追踪记录请求从开始到结束的完整路径：

```
用户请求
  → LLM 推理（1.2 秒）
    → 工具调用：lookup_order（0.3 秒）
      → 数据库查询（0.1 秒）
    → LLM 生成回复（0.8 秒）
  → 返回给用户
总延迟：2.4 秒
```

使用 OpenTelemetry 等标准协议。

### 指标（聚合数据）

| 指标 | 当前值 | 告警阈值 |
|------|--------|----------|
| P50 延迟 | 2.1 秒 | > 5 秒 |
| P99 延迟 | 12.3 秒 | > 30 秒 |
| 任务完成率 | 94% | < 85% |
| 平均 token 消耗 | 4,500 | > 10,000 |
| 工具错误率 | 2.1% | > 5% |
| 用户满意度 | 4.2/5 | < 3.5/5 |

## 构建评估体系

### 第 1 步：创建黄金测试集

收集 50-100 个代表性的测试用例，覆盖：

- 常见场景（60%）
- 边缘案例（20%）
- 对抗性输入（10%）
- 多步骤复杂任务（10%）

每个测试用例包含：输入、期望输出、评判标准。

### 第 2 步：自动化检查

对可直接验证的检查项编写自动化测试：

```python
def test_order_lookup(agent):
    result = agent.run("查一下订单 #12345")
    assert "12345" in result
    assert "已发货" in result or "处理中" in result
```

### 第 3 步：LLM 作为裁判

对需要主观判断的检查项使用 LLM 评分：

```python
def test_response_quality(agent, test_case):
    result = agent.run(test_case.input)
    score = llm_judge.evaluate(
        question=test_case.input,
        response=result,
        criteria=["accuracy", "completeness", "helpfulness"]
    )
    assert score >= 3.5
```

### 第 4 步：集成 CI/CD

```
代码变更
  → 运行单元测试
  → 运行黄金测试集（子集，5 分钟内完成）
  → 如果通过 → 合并
  → 合并后 → 运行完整评估套件（30 分钟）
  → 如果评分达标 → 部署到生产
```

## 六大常见错误

### 1. 只测"快乐路径"

只测试理想场景，不测试边缘案例和错误处理。

### 2. 没有黄金测试集

没有一组固定的测试用例来持续衡量质量。

### 3. 只看系统指标不看质量指标

关注延迟和错误率，但不关心答案是否正确。

### 4. 评估不自动化

手动跑评估 → 评估频率低 → 质量问题延迟发现。

### 5. 不追踪变化

不知道哪次改动导致质量上升或下降。每次评估结果都应该记录和对比。

### 6. 没有用户反馈闭环

不收集真实用户的反馈来改进评估标准。

---

## 核心要点

1. **评估是必需的，不能靠感觉。** 没有数据的改进是盲目迭代。

2. **四大支柱：有效性、效率、鲁棒性、安全性。** 缺一不可。

3. **两级评估：黑盒（端到端）+ 玻璃盒（轨迹分析）。** 两者互补。

4. **LLM-as-Judge 是高效的自动评估方法。** 成对比较优于绝对评分。

5. **黄金测试集是评估基石。** 50-100 个覆盖常见和边缘场景的测试用例。

6. **可观测性三支柱：日志、追踪、指标。** 没有可观测性就无法调试。

7. **质量飞轮持续运转。** 定义→测量→评估→改进→重复。
