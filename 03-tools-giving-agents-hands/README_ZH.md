# 第三课：工具——给智能体一双手

## 简介

在第一课中，我们说智能体 = 大脑 + 双手 + 记忆。第二课深入研究了大脑（LLM）。这一课我们来看双手——工具。

没有工具，智能体只是聊天机器人。它可以从训练数据中说话，但无法查询数据库、搜索网络、调用 API 或运行代码。工具使智能体能够与世界交互。

---

## ELI5：工具就像厨房设备

想象一位专业厨师。他知道所有烹饪技法——煎炒炖烤，理解风味搭配，能从头到尾构思一道菜。但如果厨房空无一物——没有炉灶、没有刀具、没有烤盘——他什么也做不出来。

知识本身不能烹饪食物。他需要工具来施展拳脚。

LLM 同理。它拥有大量知识，但没有工具，它只能说话。工具让它能查询实时数据、执行计算、发送消息——把思想变成行动。

---

## 工具的本质

工具的核心概念很简单：**函数调用**。

你定义一个函数，包含名称、描述和参数。智能体决定何时以及如何调用这个函数。你的代码执行函数并将结果返回给智能体。

```python
def get_weather(location: str, unit: str = "celsius") -> dict:
    """获取指定位置的当前天气。"""
    # ... 实现 ...
    return {"temperature": 18, "condition": "rainy", "location": location}
```

智能体看到这个函数定义后，能理解它可以做什么、需要什么参数，以及能返回什么。当用户的请求需要天气数据时，智能体就会决定调用这个函数。

### 关键洞察：模型不执行工具

这点至关重要：**LLM 不执行工具**。它只决定调用哪个工具以及使用什么参数。**你的代码**执行实际函数。

```
用户："东京的天气怎么样？"
     ↓
LLM 思考："我需要天气数据。get_weather 函数能做这个。
          参数应该是 location='Tokyo'。"
     ↓
LLM 输出：函数调用 get_weather(location="Tokyo")
     ↓
你的代码：执行 get_weather("Tokyo") → 返回 {"temperature": 18, ...}
     ↓
LLM 接收结果并格式化回复给用户
```

这个分离很重要，原因如下：
- **安全性。** 模型不能直接访问你的数据库或 API。你的代码控制实际执行内容。
- **沙箱。** 你可以在受控环境中运行工具，带有速率限制、超时和权限。
- **灵活性。** 你可以随时交换底层实现，只要函数签名保持不变。

## 工具类型

### 函数工具（自定义工具）

这是最常见的类型。你编写函数，智能体调用它们。

```python
def lookup_order(order_id: str) -> dict:
    """根据订单 ID 查找订单，返回状态、商品和配送信息。"""
    ...

def initiate_return(order_id: str, reason: str) -> str:
    """为指定订单启动退货流程。返回确认信息。"""
    ...

def search_products(query: str, limit: int = 10) -> list:
    """搜索产品目录，返回匹配的商品。"""
    ...
```

函数工具适合：
- 查询你的数据库
- 调用内部 API
- 执行自定义业务逻辑
- 与遗留系统集成

### 内置工具

一些模型提供内置工具——模型自身理解的能力，不需要你编写函数。

**Google Search（ grounding ）**
模型可以搜索网络获取最新信息。不是依赖训练数据，而是检索实时网页内容。

**代码执行**
模型可以编写并执行 Python 代码。适合数学计算、数据分析、数据转换——任何需要确定答案而不只是预测的任务。

**URL 上下文**
模型可以获取并阅读网页内容。不是盲目链接，而是实际读取页面并基于其内容回答。

**何时使用内置工具 vs. 自定义函数：**

| 场景 | 使用 |
|------|------|
| 实时网络信息 | Google Search |
| 数学计算、数据分析 | 代码执行 |
| 读取网页内容 | URL 上下文 |
| 你的数据/系统 | 自定义函数工具 |

### 智能体作为工具

你可以将整个智能体封装为另一个智能体的工具。这就是**智能体即工具**模式。

```
研究智能体（作为工具）
  ├── 搜索网络
  ├── 阅读文章
  └── 综合发现

写作智能体（作为工具）
  ├── 起草内容
  ├── 审查语法
  └── 格式化输出

编排智能体（根智能体）
  ├── 调用研究智能体获取信息
  ├── 调用写作智能体生成内容
  └── 交付最终结果
```

这种模式在智能体变得复杂时很有用。与其把所有工具堆给一个智能体，不如创建专门的子智能体，每个负责特定领域。

我们在[第七课](../07-multi-agent-systems/README_ZH.md)中深入讨论这种模式。

## 函数调用流程

以下是完整的函数调用生命周期：

```
第 1 步：定义工具
  你向模型提供函数定义（名称、描述、参数模式）。
  这些定义作为系统上下文的一部分发送。

第 2 步：用户输入
  用户发送消息："东京天气怎么样？"

第 3 步：模型决策
  模型分析请求，意识到需要外部数据。
  它匹配 get_weather 工具的描述。
  模型输出结构化的函数调用：get_weather(location="Tokyo")

第 4 步：工具执行
  你的代码拦截函数调用，执行 get_weather("Tokyo")，
  获取结果：{"temperature": 18, "condition": "rainy"}

第 5 步：结果返回
  结果作为新消息附加到上下文并发送回模型。

第 6 步：最终回复
  模型综合工具结果，生成自然语言回复：
  "东京目前气温 18 摄氏度，正在下雨。"
```

关键理解：步骤 3 到 6 是循环。模型可以在回复最终答案之前调用多个工具。

## 工具设计最佳实践

设计好的工具与设计好的 API 类似。清晰的接口、单一职责、有意义的错误。

### 1. 清晰、具体的名称

```
好： get_order_status(order_id)
差： handle_order_stuff(action)
```

名称应该让模型一目了然这个工具做什么。模型根据名称和描述做工具选择决策。

### 2. 单一职责

每个工具应该做一件事并做好。不要创建做多个事情的"瑞士军刀"函数。

```
好： create_user(), update_user(), delete_user()
差： manage_user(action, data)  // 模型必须搞清楚 action 是什么
```

### 3. 描述性的参数描述

参数描述应该告诉模型每个参数用于什么、什么值有效，以及常见示例。

```python
def query_metrics(
    service: str,      # "要查询的服务，如 'auth-service', 'api-gateway'"
    metric: str,       # "指标名称，如 'latency_p99', 'error_rate', 'cpu_usage'"
    start_time: str,   # "ISO 8601 格式的开始时间，如 '2024-03-15T10:00:00Z'"
    end_time: str,     # "ISO 8601 格式的结束时间"
    granularity: str = "5m"  # "数据点间隔，如 '1m', '5m', '1h'"
) -> dict:
    """从 Prometheus 查询服务指标。"""
```

### 4. 简洁但信息丰富的输出

工具输出需要足够信息让模型理解情况，但不能淹没上下文窗口。

```
好： {"status": "shipped", "carrier": "FedEx", "tracking": "1Z999AA10123456784", "eta": "2024-03-20"}
差： {完整的数据库行，包含 40 个字段，内部 ID 和时间戳}
```

### 5. 清晰的错误信息

当工具失败时，错误信息应该告诉模型发生了什么以及可以尝试什么。

```
好： "订单 #99999 未找到。请验证订单 ID 并重试。"
差： "内部服务器错误 500"
```

好的错误信息让智能体可以采取纠正措施——重试不同参数、尝试替代方案或向用户解释情况。

## N×M 问题与 MCP

当你有 N 个智能体和 M 个工具时，每个工具需要定制集成到每个智能体。这是 N×M 的集成负担。

```
智能体 A ──→ 工具 1（定制集成）
智能体 A ──→ 工具 2（定制集成）
智能体 B ──→ 工具 1（重写集成）
智能体 B ──→ 工具 2（重写集成）
...
```

[模型上下文协议（MCP）](../14-agent-protocols-mcp-and-a2a/README_ZH.md)通过标准化工具接口解决这个问题。智能体通过统一协议发现和使用工具，而不是定制集成。类似于 USB-C 替代各种专有连接器。

我们在[第十四课](../14-agent-protocols-mcp-and-a2a/README_ZH.md)和[第十六课](../16-mcp-deep-dive/README_ZH.md)中深入讨论 MCP。

## 常见陷阱

### 工具过多

有 50 个工具的智能体比有 5 个工具的智能体更难做出正确的工具选择。每个额外工具增加了模型选错的几率。从少量精心设计工具开始，按需添加。

### 模糊的工具描述

如果描述不清晰，模型可能：
- 完全忽略工具
- 在不应该使用时使用它
- 传入错误参数

花时间写好每个描述。具体说明工具做什么、何时使用、什么场景不适用。

### 包装过薄的工具

不要只把 API 端点直接暴露为工具。添加智能体友好的描述、参数验证和错误格式化。

```
差： 直接转发原始 API 响应（1000 行 JSON）
好：  API 响应的提取和格式化版本（10 行关键信息）
```

### 缺失错误处理

工具总会失败。网络会超时，数据库会断开连接，输入会无效。确保工具返回有用的错误信息，而不是堆栈跟踪。

### 返回过多数据

模型上下文窗口有限。返回 100 个字段当模型只需要 3 个字段时，会浪费 token 并让模型更难找到重要信息。

### 没有超时

工具调用应该有超时。没有超时，卡住的工具调用可以让整个智能体循环挂起。

## 工具设计检查清单

| 检查项 | 问题 |
|--------|------|
| 名称 | 模型能一眼理解这个工具做什么吗？ |
| 描述 | 描述是否说明何时使用及不适用场景？ |
| 参数 | 每个参数有清晰的类型、描述和示例吗？ |
| 输出 | 输出是否简洁且相关？ |
| 错误 | 失败信息是否可操作？ |
| 超时 | 工具调用有超时吗？ |
| 权限 | 工具是否遵循最小权限原则？ |
| 幂等性 | 重复调用会造成损害吗？ |

## 综合运用：调查智能体

以下是工具设计在实际智能体中的应用示例。

**场景：** DevOps 调查智能体，帮助排查生产问题。

**工具集：**

```python
def get_alert_details(alert_id: str) -> dict:
    """获取指定告警的详细信息，包括严重性、触发条件和受影响服务。"""

def query_metrics(
    service: str,
    metric: str,
    start_time: str,
    end_time: str
) -> dict:
    """从 Prometheus 查询指定服务的指标数据。"""

def search_logs(
    service: str,
    query: str,
    start_time: str,
    end_time: str,
    limit: int = 100
) -> list:
    """搜索指定服务的日志条目，支持过滤和关键词匹配。"""

def list_recent_deployments(service: str, limit: int = 5) -> list:
    """列出指定服务的最近部署，包含版本、时间和部署者。"""

def get_runbook(alert_type: str) -> str:
    """获取指定告警类型的运行手册（标准排查步骤）。"""

def create_incident(
    summary: str,
    severity: str,
    affected_services: list,
    initial_findings: str
) -> str:
    """创建新事件并返回事件 ID。"""
```

**智能体工作流：**

```
1. 收到告警 → get_alert_details(alert_id)
2. 获取运行手册 → get_runbook(alert_type)
3. 查询相关指标 → query_metrics(service, metric, ...)
4. 搜索日志异常 → search_logs(service, query, ...)
5. 检查最近部署 → list_recent_deployments(service)
6. 综合发现，创建事件或建议修复
```

每个工具都有单一职责、清晰的输入输出，以及可操作的错误信息。智能体可以按顺序使用它们，或根据发现跳转到不同工具。

## 核心要点

1. **工具是智能体与世界交互的方式。** 没有工具，智能体只是聊天机器人。

2. **模型不执行工具——它只决定调用什么。** 你的代码执行实际函数，控制安全性和权限。

3. **好的工具设计对智能体性能至关重要。** 清晰的名称、单一职责、描述性参数、简洁输出和可操作的错误信息。

4. **从少量工具开始。** 3-5 个精心设计工具通常优于 50 个设计糟糕的工具。

5. **工具失败是必然的。** 设计可操作的错误信息、超时和重试逻辑。

6. **N×M 集成问题通过 MCP 等标准协议解决。** 统一接口减少集成负担。

7. **智能体本身可以作为工具。** 子智能体模式在复杂场景中通过专业化提高效果。
