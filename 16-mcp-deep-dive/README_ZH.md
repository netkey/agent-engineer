# 第十六课：MCP 深度解析

## 简介

第十二课和第十四课介绍了 MCP 是什么。本课深入底层：MCP 架构、传输方式、与 CLI 的对比、安全考量以及何时该用（或不该用）MCP。

---

## ELI5：排插

没有排插前，每个电器需要单独插到墙上插座。有了排插，所有电器插到排插上，排插只占一个墙上插座。

MCP 就是智能体的排插。不是每个工具各写一套集成代码——通过 MCP，所有工具走一个标准接口。

---

## MCP 架构回顾

```
Host（宿主应用）
  │
  ├── Client（MCP 客户端）
  │     │
  │     ├── stdio（本地进程间通信）
  │     └── Streamable HTTP（远程服务调用）
  │
  └── Server（MCP 服务器）
        │
        ├── Tools：可调用的函数
        ├── Resources：可读取的数据
        └── Prompts：可复用的提示模板
```

**协议：** JSON-RPC 2.0

### 传输方式

| 传输 | 用途 | 适合场景 |
|------|------|----------|
| **stdio** | 本地，通过 stdin/stdout 子进程 | 本地工具、桌面应用 |
| **Streamable HTTP** | 远程，HTTP 连接 | 远程服务、多用户环境 |

> 注：Streamable HTTP 已替代已废弃的 SSE 传输。

### 原语采用率

- **Tools（工具）**：99% 的 MCP 服务器实现
- **Resources（资源）**：30-35%
- **Prompts（提示）**：约 30%

## MCP vs. CLI：永恒的争论

### CLI 方案的优势

| 方面 | CLI | MCP |
|------|-----|-----|
| Token 消耗 | 低（约 1,400 token） | 高（约 44,000 token，32 倍） |
| 可靠性 | 100%（确定性） | 约 72% |
| 上下文窗口 | 不占用 | 工具描述占用 |
| 开发成本 | 写个脚本就行 | 需要 MCP 服务器框架 |
| 调试 | 直接跑命令看输出 | 需要通过客户端调试 |

### MCP 方案的优势

| 方面 | MCP | CLI |
|------|-----|-----|
| 安全/认证 | OAuth 2.1 with PKCE | 有限 |
| 多用户环境 | 适合 | 不适合 |
| 工具发现 | 自动列出和描述 | 需要手动文档 |
| 结构化 I/O | JSON Schema 定义 | 解析 stdout 容易出错 |
| 一次编写到处用 | 所有 MCP 客户端可用 | 每个工具需单独集成 |

### 何时用哪个

| 场景 | 推荐 | 原因 |
|------|------|------|
| 个人脚本 | CLI | 简单快速 |
| 一次性任务 | CLI | 不需要基础设施 |
| 团队共享工具 | MCP | 标准化，多人可用 |
| 需要认证 | MCP | 内建 OAuth |
| 工具发现重要 | MCP | 自动列出能力 |
| 延迟敏感 | CLI | 无网络开销 |
| 上下文受限 | CLI | 不占用 token |

### mcp2cli

[mcp2cli](https://github.com/modelcontextprotocol/mcp2cli) 将 MCP 服务器转换为 CLI 工具，减少 96-99% 的 token 消耗，同时保留 MCP 的结构化能力。

## OWASP MCP Top 10

| 排名 | 威胁 | 描述 |
|------|------|------|
| 1 | **工具投毒（Tool Poisoning）** | 恶意工具返回有害结果诱导智能体 |
| 2 | **工具影子（Tool Shadowing）** | MCP 工具与内置工具功能重叠，导致选择冲突 |
| 3 | **过度权限（Excessive Permissions）** | 工具拥有超出所需的访问权限 |
| 4 | **上下文窗口膨胀（Context Window Bloat）** | 过多工具描述填满上下文窗口 |
| 5 | **密钥泄露（Secret Exposure）** | 工具实现中硬编码的凭证被暴露 |
| 6 | **输入注入（Input Injection）** | 恶意输入通过 MCP 工具传入 |
| 7 | **供应链攻击（Supply Chain Attacks）** | 第三方 MCP 服务器被篡改 |
| 8 | **拒绝服务（DoS）** | 工具调用被滥用消耗资源 |
| 9 | **数据外泄（Data Exfiltration）** | 工具被诱导发送数据到恶意端点 |
| 10 | **权限提升（Privilege Escalation）** | 工具被用于执行超出权限的操作 |

### MCP 安全检查清单

- [ ] 审查所有 MCP 服务器代码
- [ ] 最小权限原则——工具只有所需的最小访问权限
- [ ] 输入验证——所有输入在服务端验证
- [ ] 输出验证——智能体在使用前验证工具结果
- [ ] 认证和授权——使用 OAuth 保护远程服务器
- [ ] 审计日志——记录所有工具调用
- [ ] 速率限制——防止滥用
- [ ] 网络隔离——工具服务器在受限网络中运行

## 构建 MCP 服务器的决策

### 传输选择

- **stdio**：本地工具、桌面应用、不需要网络访问
- **Streamable HTTP**：远程服务、需要多用户访问、需要认证

### 工具粒度

每个服务器 5-20 个工具为宜。太多工具导致：
- 上下文窗口膨胀
- 模型选择困难
- 维护负担

### 命名和描述

```
好：search_knowledge_base(query, filters)
    "在内部知识库中搜索相关文档。支持关键词和过滤器。"
差：search_stuff(thing)
    "搜索东西"
```

### 输出设计

工具输出应该：
- 简洁但信息丰富
- 有结构化格式（JSON）
- 包含错误信息而非堆栈跟踪
- 只返回调用者需要的字段

## 决策树：CLI vs. MCP

```
需要给智能体加工具吗？
  │
  +-- 是 --> 需要认证或多用户吗？
  │            │
  │            +-- 是 --> 使用 MCP
  │            │
  │            +-- 否 --> 需要工具发现或结构化 I/O 吗？
  │                         │
  │                         +-- 是 --> 使用 MCP
  │                         │
  │                         +-- 否 --> 上下文窗口受限吗？
  │                                      │
  │                                      +-- 是 --> 使用 CLI
  │                                      │
  │                                      +-- 否 --> CLI 或 MCP 都行
  │                                                （优先 CLI 更简单）
```

## 核心要点

1. **MCP 是标准化工具接口。** 解决 N×M 集成问题，一次编写到处使用。

2. **MCP 不是银弹。** CLI 在简单场景下更省 token、更可靠。根据场景选择。

3. **Token 消耗是关键成本。** MCP 工具描述可能比 CLI 多用 32 倍 token。考虑 mcp2cli 减少消耗。

4. **安全必须纵深防御。** OWASP MCP Top 10 列出了主要威胁——每层都需要防护。

5. **工具粒度影响效果。** 每个服务器 5-20 个工具为宜。太多导致上下文膨胀和选择困难。

6. **好的命名和描述决定工具选择准确率。** 模型靠这些决定用哪个工具。
