# 第七课：多智能体系统——当单个智能体不够用时

## 简介

一个智能体能做很多事，但就像一个人无法同时做好所有工作一样，复杂任务通常需要专业分工。

多智能体系统让不同智能体各司其职、协同工作——就像医院里有医生、护士、检验科和药房，每个角色专注不同领域，共同为患者提供服务。

本课介绍为什么需要多智能体、不同协作架构、通信模式、角色分工以及协调挑战。

---

## ELI5：杂货店与百货商场

一家街角杂货店由一个人经营——他进货、收银、理货、打扫卫生。能运转，但规模有限。

百货商场有专门部门：服装区、食品区、电器区、收银台、客服。每个部门有专人负责，能处理更多客户和更复杂需求。

单智能体 = 杂货店。适合简单、小规模任务。
多智能体 = 百货商场。适合复杂、大规模、需要专业知识的任务。

## 单智能体的局限

| 局限 | 描述 |
|------|------|
| **上下文窗口限制** | 单个智能体的记忆有限，无法同时追踪多个独立任务 |
| **工具冲突** | 太多工具导致模型选择困难，降低准确性 |
| **专业知识稀释** | 通用智能体在每个领域都"还行"，但不如专精智能体 |
| **不可并行化** | 单智能体顺序执行，无法同时处理独立任务 |
| **调试困难** | 一个复杂智能体出错时，很难定位问题在哪个环节 |

## 多智能体的优势

| 优势 | 描述 |
|------|------|
| **专业分工** | 每个智能体专注于特定领域，质量更高 |
| **并行执行** | 独立任务可同时处理，降低延迟 |
| **上下文隔离** | 每个智能体管理自己的上下文，避免信息混杂 |
| **质量门控** | 一个智能体的输出可作为另一个智能体的输入，形成检查机制 |
| **可扩展性** | 按需添加新智能体，不影响已有系统 |

## 多智能体架构

### 1. 顺序架构（流水线）

智能体按顺序排列，每个处理上一步的输出。

```
用户请求 → [理解智能体] → [研究智能体] → [写作智能体] → 最终输出
```

**适用场景：** 任务有清晰的阶段划分，每步依赖上一步结果。
**示例：** 内容生成流水线——研究→起草→编辑→发布。

### 2. 层级架构（组织树）

一个协调智能体将任务分解并分配给子智能体。

```
         [协调智能体]
        /     |      \
   [研究]  [编码]  [测试]
     |        |        |
  搜索结果   代码     测试报告
     \        |        /
      \       |       /
       [协调智能体整合结果]
```

**适用场景：** 复杂项目需要中央协调。
**示例：** 软件开发——项目经理（协调）分配任务给开发、测试、文档编写智能体。

### 3. 协作架构（头脑风暴）

多个智能体平等协作，共同解决问题。

```
[智能体 A] ←→ [智能体 B] ←→ [智能体 C]
     ↖_________________________↙
          共享讨论区
```

**适用场景：** 需要多角度观点的问题。
**示例：** 架构设计评审——安全智能体、性能智能体、可维护性智能体各自发表意见。

### 4. 竞争架构（竞标）

多个智能体独立产生方案，由评估者选择最佳方案。

```
                 [任务]
                /   |   \
         [智能体 A] [智能体 B] [智能体 C]
                \   |   /
              [评估智能体]
                    |
               [最佳方案]
```

**适用场景：** 创意任务或需要探索多种可能性的问题。
**示例：** 广告文案生成——三个智能体各写一版，评估者选最好的。

## 通信模式

智能体之间如何传递信息？

### 1. 直接消息

一个智能体直接向另一个智能体发送消息。

```
[智能体 A] --"搜索结果如下"--> [智能体 B]
```

简单直接，适合顺序和层级架构。

### 2. 共享黑板

所有智能体读写共享存储空间。

```
[智能体 A] ↘
             [共享黑板] ← 所有智能体可读写
[智能体 B] ↗
[智能体 C] ↗
```

适合协作架构，智能体异步贡献信息。

### 3. 基于事件

智能体发布和订阅事件。

```
[智能体 A] --发布: "研究完成"--> [事件总线]
                                         ↓
                            [智能体 B] 订阅并接收事件
```

松耦合，适合复杂、动态的系统。

## 智能体角色

在多智能体系统中，常见的角色分工：

### 规划者（Planner）

**职责：** 分解任务、制定计划、分配工作。
**能力：** 高层推理、优先级判断、资源分配。
**不做：** 执行具体任务。

```
用户："为我们的新产品制定营销方案"
规划者："需要三步：1) 市场调研 2) 竞品分析 3) 方案撰写。
        交给调研智能体做 1 和 2，写作智能体做 3。"
```

### 检索者（Retriever）

**职责：** 收集信息——搜索、查询、提取。
**能力：** 工具使用、数据提取、信息综合。
**不做：** 决策或判断。

```
规划者："查找竞品定价信息"
检索者：[搜索网络] [查询数据库] [提取数据]
       → "竞品 A 定价 $99/月，竞品 B 定价 $149/月..."
```

### 执行者（Executor）

**职责：** 执行具体行动——写代码、发邮件、部署。
**能力：** 工具使用、操作执行、结果验证。
**不做：** 规划或研究。

```
规划者："部署 v2.3 到生产环境"
执行者：[运行部署脚本] [验证健康检查] [报告结果]
       → "部署成功，所有健康检查通过"
```

### 评估者（Evaluator）

**职责：** 审查输出质量、验证正确性、提供反馈。
**能力：** 批判性分析、标准比对、反馈生成。
**不做：** 原始生成或执行。

```
执行者：[生成代码]
评估者：[审查代码] → "第 42 行有潜在 SQL 注入，
                      缺少错误处理，整体逻辑正确。
                      建议修复后重新提交。"
```

## 综合运用：客户退款系统

以下是一个真实场景中多智能体如何协作的示例。

**场景：** 客户请求退款，需要多个系统协作。

```
[用户]："我想退款，订单号 #78901"

[路由智能体]：识别意图为"退款请求"，
              分配给退款处理工作流

[解决智能体]：了解用户需求，
              "客户想退款 #78901。先查订单状态。"

[客户智能体]：查询客户历史记录，
              "客户 Jane，下单 30 天，购买过 12 次，
               之前无退款，信誉良好。"

[合规智能体]：检查退款政策，
              "订单 30 天内可全额退款。
               客户符合退款条件。批准退款。"

[支付智能体]：处理退款，
              "退款 $149.99 到原支付方式。
               预计 3-5 个工作日到账。确认号：REF-12345。"

[解决智能体]：综合结果，回复用户，
              "Jane，您的退款 $149.99 已处理。
               预计 3-5 个工作日到账。确认号：REF-12345。"
```

每个智能体只做自己擅长的事，协调智能体整合结果。

## 协调挑战

多智能体系统比单智能体复杂得多。主要挑战：

### 1. 死锁

两个智能体互相等待对方先行动。

```
智能体 A："等智能体 B 提供数据才能继续"
智能体 B："等智能体 A 确认后才能提供数据"
→ 两者都卡住
```

**修复：** 定义明确的依赖顺序，或设置超时后自动推进。

### 2. 循环委派

智能体之间来回传递任务。

```
智能体 A："这个问题交给智能体 B 处理"
智能体 B："这个问题应该由智能体 A 处理"
智能体 A："交给智能体 B..."
→ 无限循环
```

**修复：** 设置最大委派次数，超过后升级处理。

### 3. 冲突行动

两个智能体采取互相矛盾的行动。

```
智能体 A："扩容到 10 台服务器"
智能体 B："缩容到 2 台服务器节约成本"
→ 同时执行，结果不可预测
```

**修复：** 中央协调者仲裁，或定义互斥锁防止冲突操作。

### 4. 资源竞争

多个智能体同时访问同一资源。

```
智能体 A 和 B 同时更新同一数据库记录
→ 数据不一致
```

**修复：** 资源锁定、队列或版本控制。

### 5. 状态不一致

智能体对同一事实有不同理解。

```
智能体 A 认为"订单已退款"
智能体 B 认为"订单待退款"
→ 因为通信延迟导致信息不同步
```

**修复：** 共享事实来源（共享黑板或中央状态存储）。

## 设计原则

### 1. 从简单开始

在添加多智能体复杂性之前，先尝试单智能体。只有当以下情况出现时才拆分：
- 工具太多导致选择混乱
- 任务天然可分为独立子任务
- 需要并行执行
- 需要质量门控（生成者+审查者）

### 2. 定义清晰边界

每个智能体应该有明确的职责范围。模糊的边界导致冲突和遗漏。

```
清晰："研究智能体只负责信息收集，不做决策"
模糊："研究智能体负责研究和相关决策"
```

### 3. 为失败设计

多智能体系统中，一个智能体失败可能影响整个流程。设计容错：
- 超时后自动重试
- 备用智能体或降级路径
- 失败时优雅地向用户解释

### 4. 使通信可观测

记录智能体之间的所有消息。调试多智能体系统时，消息日志是最有价值的排障工具。

### 5. 限制自主性

给智能体设定自主行动的边界。超出边界需要人工批准。

```
允许：智能体可以批准 $100 以内的退款
需批准：$100-$500 的退款需人工审核
禁止：超过 $500 的退款不允许自动处理
```

### 6. 人工作为断路器

在高影响决策上，始终保留人工介入的能力。智能体建议，人类决定。

## 核心要点

1. **多智能体解决单智能体的规模和专业化限制。** 分工带来更高质量和更好的并行性。

2. **四种基本架构：** 顺序（流水线）、层级（组织树）、协作（头脑风暴）、竞争（竞标）。

3. **通信模式决定松耦合程度。** 直接消息最简单，事件总线最灵活。

4. **明确角色分工。** 规划者、检索者、执行者、评估者各司其职。

5. **协调是最大挑战。** 死锁、循环委派、冲突行动、资源竞争——都需要设计时预防。

6. **从简单开始。** 只有明确理由时才引入多智能体。复杂性是真实成本。

7. **人工是最终的安全网。** 对于高影响决策，智能体建议、人类决定。
