# 第十七课：智能体技能——打包可复用的领域专业知识

## 简介

工具给智能体双手——让它能行动。技能给智能体知识——让它知道**怎么做**。

如果把工具比作扳手和螺丝刀，技能就是修理手册。扳手让你拧螺丝，手册告诉你**按什么顺序、拧多大扭矩、遇到锈蚀怎么办**。

本课介绍智能体技能是什么、如何定义、与工具和 MCP 的区别，以及何时使用。

---

## ELI5：扳手 vs. 修理手册 / 食谱卡

**工具 = 扳手。** 它能拧螺母，但不知道用在哪、什么时候用、出问题怎么办。

**技能 = 修理手册。** 它告诉智能体："遇到 X 情况，用 Y 方法，按 Z 步骤，注意 W 陷阱。"

**厨房类比：**
- **工具** = 锅碗瓢盆（做事的设备）
- **技能** = 食谱卡（做某道菜的步骤、配料、技巧）

没有食谱卡，即使有全套厨具，你也不知道怎么做宫保鸡丁。

## 为什么需要技能

### 问题：上下文窗口有限

把所有知识都塞进系统提示？15 万字的专业知识占满 100 万 token 的上下文，留给对话和工具结果的空间就少了。

### 解决：按需加载

技能系统采用**渐进式披露**——先只加载轻量元数据（约 100 token），需要时才加载完整指令。

```
不使用技能：  系统提示 + 所有知识 = ~150,000 token（始终加载）
使用技能：    系统提示 + 元数据 = ~2,000 token（按需加载完整内容）
```

### 额外好处

- **独立迭代** —— 技能可以独立更新，不需要改智能体代码
- **跨团队共享** —— 一个团队编写的技能，其他团队直接使用
- **关注点分离** —— 领域专家写技能，工程师写智能体逻辑

## SKILL.md 格式

### YAML 前置元数据

```yaml
---
name: code-review
description: 全面的代码审查流程，包括安全性、测试、命名和文档检查
license: Apache-2.0
compatibility:
  min_adk_version: "1.0.0"
metadata:
  author: "平台工程团队"
  version: "2.1.0"
---
```

### Markdown 正文（L2 指令）

```markdown
## 代码审查步骤

### 1. 安全检查
- 检查 SQL 注入、XSS、命令注入
- 验证输入是否经过过滤和转义
- 确认没有硬编码的密钥或凭证

### 2. 测试覆盖
- 新代码有对应的单元测试吗？
- 边界条件和边缘案例被覆盖了吗？
- 集成测试验证了端到端流程吗？

### 3. 命名约定
- 变量和函数使用描述性名称
- 函数/变量：snake_case，类：CamelCase
- 常量：UPPER_CASE

### 4. 文档
- 公共 API 有文档字符串吗？
- 复杂的逻辑有注释解释"为什么"（不是"做什么"）
- README 更新了吗？
```

## 渐进式披露

技能分三层加载：

| 层级 | 内容 | Token 消耗 | 何时加载 |
|------|------|-----------|----------|
| **L1 元数据** | 名称、描述、版本 | ~100 | 始终——用于决定是否激活 |
| **L2 指令** | 详细步骤、规则、决策点 | 2,000-5,000 | 当元数据匹配任务时 |
| **L3 资源** | 参考文档、脚本、模板 | 可变 | 当指令引用具体资源时 |

**工作流程：**
```
智能体收到任务
  → 扫描所有技能的 L1 元数据
  → 匹配到相关技能？
    → 是：加载 L2 指令
    → 否：跳过
  → L2 指令需要参考文档？
    → 是：加载 L3 资源
    → 否：仅用 L2
```

## 技能 vs. 工具 vs. MCP

| 方面 | 技能（Skill） | 工具（Tool） | MCP |
|------|--------------|-------------|-----|
| **本质** | 知识 | 行动 | 协议 |
| **类比** | 食谱卡 | 锅碗瓢盆 | USB 接口 |
| **作用** | 告诉智能体"怎么做" | 让智能体"能做什么" | 标准化工具连接方式 |
| **格式** | SKILL.md（Markdown） | 函数定义 | JSON-RPC |
| **加载时机** | 按需（渐进式披露） | 始终在上下文窗口中 | 按需发现 |
| **消耗** | 加载时占用上下文 | 始终占用上下文 | 发现时占用上下文 |

**简单区分：**
- **技能** = 知识（你知道该怎么做）
- **工具** = 行动（你有能力去做）
- **MCP** = 协议（标准连接方式）

## 编写指南

### 1. 一个技能专注一个领域

```
好：code-review, database-migration, incident-response
差：everything（把所有东西塞一个技能）
```

### 2. 为 LLM 写作

- 用清晰的步骤，不假设常识
- 包含决策点："如果 X，做 Y；否则做 Z"
- 展示期望输出

### 3. 保持 L2 在 5,000 token 以内

超过这个长度考虑拆分为多个技能。

### 4. 包含决策点

```
如果代码涉及用户输入：
  → 执行完整安全检查
否则：
  → 跳过安全检查，继续测试覆盖
```

### 5. 展示期望输出

```
审查报告格式：
  ## 审查结果
  - 安全性：通过/发现问题 X
  - 测试：覆盖率 Y%，缺少 Z 场景
  - 命名：符合/不符合约定
  - 文档：完整/缺少 W 部分
```

## ADK 中的技能

ADK 提供 `SkillToolset` 加载技能：

```python
from adk.tools import SkillToolset

# 从 skills/ 目录加载
skills = SkillToolset(root_dir="skills/")
agent.tools = [skills]
```

ADK 支持两种技能定义方式：
- **文件式** —— `skills/` 目录下的 SKILL.md 文件
- **代码式** —— 在代码中定义技能

## 实际示例

### 数据库迁移技能

```yaml
---
name: database-migration
description: 使用 Alembic 管理数据库迁移的完整流程
---
```

```markdown
## 数据库迁移步骤

### 创建新迁移
1. 运行 `alembic revision --autogenerate -m "描述"`
2. 审查生成的迁移文件
3. 手动编辑需要调整的变更

### 应用迁移
1. 运行 `alembic upgrade head`
2. 验证迁移成功
3. 如果有数据迁移，单独运行数据脚本

### 回滚
1. 运行 `alembic downgrade -1`
2. 验证回滚成功

## 注意事项
- 迁移必须可回滚
- 大数据量变更需要分批执行
- 生产环境迁移需要人工审批
```

### 事故响应技能

```yaml
---
name: incident-response
description: 生产事故响应流程，包括诊断、缓解和事后分析
---
```

```markdown
## 事故响应步骤

### 1. 确认事故
- 检查监控仪表板
- 验证告警不是误报

### 2. 诊断
- 查询最近 30 分钟指标：`gcloud monitoring read ...`
- 检查最近部署：`gcloud builds list --limit=5`
- 查看错误日志：`gcloud logging read ...`

### 3. 缓解
- 如果是部署引起的：回滚到上一版本
- 如果是流量激增：扩容或启用限流
- 如果是数据库：检查连接池和慢查询

### 4. 事后分析
- 记录时间线
- 确定根因
- 制定改进措施
```

## 何时使用技能

| 场景 | 推荐方案 |
|------|----------|
| 可复用的领域知识 | 技能（SKILL.md） |
| 执行具体操作 | 工具（函数） |
| 连接外部服务 | MCP 或工具 |
| 项目上下文 | AGENTS.md |

## 核心要点

1. **技能 = 打包的领域知识。** 告诉智能体"怎么做"，不只是"能做什么"。

2. **渐进式披露解决上下文限制。** L1/L2/L3 三层按需加载，节省大量 token。

3. **技能与工具互补。** 技能是知识，工具是行动。两者结合才能做好复杂任务。

4. **独立迭代是核心优势。** 领域专家更新技能，不需要改智能体代码。

5. **一个技能专注一个领域。** 保持聚焦，不超过 5,000 token。
