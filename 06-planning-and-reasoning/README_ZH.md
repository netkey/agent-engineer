# 第六课：规划与推理——智能体如何思考和决策

## 简介

给定一个复杂任务，你会怎么做？不太可能直接跳进去盲目操作。你会先想想要做什么、分解步骤、按顺序排列、然后开始执行——过程中根据发现调整计划。

智能体也是如此。规划与推理是使智能体从简单的"反应式"工具升级为能处理复杂、多步骤问题的智能系统的能力。

本课介绍智能体如何规划工作、做出推理决策，以及在计划出错时如何调整。

---

## ELI5：规划就像打包行李

想象你要去旅行。你不会把所有东西一股脑塞进箱子。你会：

1. **列出清单。** "我需要衣服、洗漱用品、充电器、证件。"
2. **分类整理。** "衣服放一层，洗漱用品放侧袋，电子产品放隔层。"
3. **按顺序打包。** "先放衣服，因为最占空间。小物件最后放。"
4. **检查并调整。** "箱子满了——把厚外套拿出来穿在身上。"

这就是规划：列出要做的事，安排顺序，执行，遇到意外时调整。智能体做同样的事情——只是处理的是信息而不是衣服。

## 智能体的问题解决循环

智能体处理复杂任务时，遵循一个循环：

```
1. 获取任务（Get Mission）
     ↓
2. 扫描当前状态（Scan Scene）
     ↓
3. 思考下一步（Think）
     ↓
4. 执行行动（Act）
     ↓
5. 观察结果（Observe）
     ↓
6. 重复或结束
```

这个循环看起来像[第四课](../04-agentic-design-patterns/README_ZH.md)中的 ReAct 模式——确实如此。规划是 ReAct 循环中"思考"部分的深化。

## 规划策略

### 计划后执行（Plan-then-Execute）

智能体在执行前先制定完整计划，然后按计划逐步执行。

```
用户："分析我们 Q1 的销售数据，找出下降原因，并提出改进建议。"

智能体计划：
  1. 查询 Q1 销售数据
  2. 按产品线和地区细分
  3. 与 Q4 数据对比
  4. 识别下降最明显的类别
  5. 查找相关因素（市场变化、竞品活动、产品问题）
  6. 综合发现
  7. 提出改进建议

执行：[按顺序执行步骤 1-7]
```

**优点：** 结构清晰，步骤可见，便于调试。
**缺点：** 如果计划基于错误假设，可能走偏。

### 反应式（Reactive）

智能体不做前期计划——逐步决策，每步基于上一步结果决定下一步。

```
用户："找出我们网站的性能问题。"

智能体：
  1. 先检查最近的监控数据 → 发现 CPU 飙高
  2. CPU 为什么飙高？→ 查看进程列表 → 发现异常查询
  3. 哪个查询？→ 检查慢查询日志 → 找到具体 SQL
  4. 为什么突然变慢？→ 检查最近部署 → 发现索引被误删
```

**优点：** 灵活适应发现，不做错误假设。
**缺点：** 可能走弯路，缺少全局视角。

### 混合方法

大多数实际智能体使用混合方法：制定粗略的高级计划，然后用反应式方式执行每个步骤。

```
高级计划：[数据收集 → 分析 → 建议]
  数据收集阶段：反应式探索（不知道会发现什么）
  分析阶段：根据收集的数据调整分析方法
  建议阶段：基于分析结果生成
```

这是最实用的方法——有方向但不僵化。

## 分层规划

复杂任务需要分层分解。

```
目标（Epic）："改进网站性能"
  │
  ├── 故事（Story）："减少页面加载时间到 2 秒内"
  │     │
  │     ├── 任务（Task）："优化图片加载"
  │     │     └── 步骤（Step）："启用 WebP 格式"
  │     │     └── 步骤（Step）："实现懒加载"
  │     │
  │     ├── 任务（Task）："减少 JavaScript 体积"
  │     │     └── 步骤（Step）："Tree-shaking"
  │     │     └── 步骤（Step）："代码分割"
  │
  ├── 故事（Story）："优化数据库查询"
        └── 任务（Task）："添加缺失索引"
        └── 任务（Task）："重写 N+1 查询"
```

分层规划使智能体能够：
- 在高层跟踪进度
- 在低层灵活调整具体步骤
- 并行化独立任务
- 在任一层次重新规划

## 重新规划

计划很少完美执行。智能体需要能重新规划。

### 重新规划的触发条件

- **意外结果。** "查询返回空——需要换搜索策略。"
- **工具失败。** "API 超时——尝试备用数据源。"
- **新信息。** "发现部署变更——这可能解释性能下降。"
- **目标变更。** "用户说现在更关心成本而非性能。"

### 重新规划策略

| 策略 | 描述 | 示例 |
|------|------|------|
| **局部调整** | 微调当前步骤 | "这个查询没结果，试试不同的日期范围" |
| **步骤插入** | 在计划中添加新步骤 | "发现新问题——先调查这个再做下一步" |
| **计划修订** | 重新制定剩余计划 | "原方法不行，换一种分析思路" |
| **目标修改** | 与用户确认是否调整目标 | "数据不足，无法做完整分析。是否缩小范围？" |

## 推理策略

### 思维链（Chain-of-Thought，CoT）

逐步推理，每步基于前一步的结论。

```
问题："服务器集群有 10 台机器，每台处理 1000 请求/秒。
      如果 20% 流量被路由到 2 台机器，这些机器过载了吗？"

思维链：
  1. 总处理能力：10 × 1000 = 10,000 请求/秒
  2. 20% 流量到 2 台机器 = 2,000 请求/秒
  3. 2 台机器的处理能力 = 2 × 1000 = 2,000 请求/秒
  4. 2,000 = 2,000 → 刚好满负荷，没有过载但无余量
  5. 但如果流量波动超过 0%，就会过载
  结论：处于临界状态，任何流量波动都会导致过载
```

### 思维树（Tree-of-Thoughts，ToT）

探索多条推理路径，评估每条路径，选择最佳的。

```
问题："API 响应时间从 200ms 增加到 2 秒。可能的原因？"

路径 A："数据库查询变慢"
  → 检查：慢查询日志
  → 结果：正常，排除

路径 B："网络延迟增加"
  → 检查：网络监控
  → 结果：正常，排除

路径 C："应用层新增逻辑导致处理变慢"
  → 检查：最近部署变更
  → 结果：发现新增了数据验证步骤，增加了 1.5 秒
  → 确认：这是根因

决策：聚焦路径 C，优化数据验证逻辑
```

### 自洽性（Self-Consistency）

对同一问题多次推理（可能用不同思路），选择最一致的答案。

```
问题："我们应该在周五部署这个变更吗？"

推理 1："是的——变更小，风险低，周末前完成能让团队安心"
推理 2："不——周五部署意味着如果出问题，周末要加班修复"
推理 3："看情况——有回滚方案可以部署，没有就应该推迟"

综合：有自动回滚 + 监控 → 可以部署
      没有回滚 → 推迟到周一
```

## 编排层的作用

编排层管理规划与推理的流程：

- **维护当前计划。** 跟踪哪些步骤已完成，哪些待执行。
- **管理上下文。** 决定哪些信息注入上下文窗口。
- **处理重新规划。** 检测计划偏差并触发重新规划。
- **强制执行约束。** 确保智能体不违反安全或业务规则。

编排层是"看不见的手"，引导智能体在规划灵活性的同时不偏离轨道。

## 常见失败模式

### 1. 无限循环

智能体反复尝试同一件事而没有进展。

```
智能体：调用 search("X") → 无结果
智能体：调用 search("X") → 无结果
智能体：调用 search("X") → 无结果
...
```

**修复：** 设置最大迭代次数。超过后强制停止或升级。

### 2. 计划漂移

智能体偏离原始目标，陷入不相关的子任务。

```
原始目标："分析销售数据"
... 30 分钟后 ...
当前任务："研究数据库索引优化方案"
```

**修复：** 定期检查当前步骤是否与原始目标对齐。如果不，重新规划。

### 3. 过度规划

智能体花太多时间制定详细计划而不是执行。

```
用户："查一下明天天气"
智能体：制定 15 步计划，包括数据源评估、格式选择、缓存策略...
```

**修复：** 根据任务复杂度调整规划深度。简单任务不需要复杂计划。

### 4. 级联失败

一个步骤失败，智能体没有备选方案，整个计划崩溃。

```
步骤 1：查询数据库 → 失败（连接超时）
步骤 2：（依赖步骤 1）→ 无法执行
步骤 3：（依赖步骤 2）→ 无法执行
整个计划停滞
```

**修复：** 为每个关键步骤设计备选方案。如果 A 失败，尝试 B。

### 5. 上下文耗尽

长计划积累的上下文填满窗口，导致智能体"忘记"早期信息。

**修复：** 使用第五课中的上下文管理策略——摘要、截断、外部化。

## 核心要点

1. **规划让智能体处理复杂任务。** 没有规划，智能体只能做单步操作。

2. **混合方法最实用。** 粗略的前期计划 + 反应式执行，兼顾方向和灵活性。

3. **分层规划处理复杂性。** 目标→故事→任务→步骤的层级使智能体能跟踪进度并在任一层次调整。

4. **重新规划是必需的，不是例外。** 计划几乎从不完全按计划执行。好的智能体能优雅地调整。

5. **推理策略影响准确性。** 思维链、思维树和自洽性都能提升复杂推理任务的准确性——代价是更多 token 和延迟。

6. **编排层是安全网。** 它防止无限循环、计划漂移和上下文耗尽。

7. **避免过度规划。** 规划深度应与任务复杂度匹配。简单任务需要简单计划。
