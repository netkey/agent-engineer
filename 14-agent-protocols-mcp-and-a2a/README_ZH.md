# 第十四课：智能体协议——MCP 与 A2A

## 简介

当你有 N 个智能体和 M 个工具，每个工具都需要定制集成到每个智能体——这是 N×M 的集成负担。你需要的是标准协议：一个通用接口，让所有工具和智能体即插即用。

本课介绍两个关键的开放标准协议：
- **MCP（Model Context Protocol）** —— 智能体与工具的通用接口
- **A2A（Agent-to-Agent Protocol）** —— 智能体之间的通信协议

---

## ELI5：电源适配器与电话

**MCP 像 USB-C 接口。** 以前每个设备有专用充电器——手机、平板、相机各有各的线。USB-C 出来了：一个接口，所有设备都能用。MCP 同理：一个协议，所有工具都能被所有智能体调用。

**A2A 像打电话。** 你知道朋友电话号码，打过去，聊天，挂断。不需要知道对方的手机是什么牌子、用什么运营商。A2A 让智能体之间以标准方式沟通——发送任务、接收结果、追踪进度。

## MCP：模型上下文协议

### 架构

```
Host（宿主应用）
  │
  ├── Client（MCP 客户端）
  │     │
  │     ├── stdio（本地进程）
  │     └── Streamable HTTP（远程服务）
  │
  └── Server（MCP 服务器）
        │
        ├── Tools（工具）
        ├── Resources（资源）
        └── Prompts（提示模板）
```

- **Host**：运行 MCP 客户端的应用（如 Claude Code、ADK 智能体）
- **Client**：发起 MCP 请求的一方
- **Server**：提供工具、资源、提示的服务端
- **传输层**：stdio（本地）或 Streamable HTTP（远程）

### 三大原语

| 原语 | 用途 | 示例 |
|------|------|------|
| **Tools（工具）** | 智能体可调用的函数 | 搜索、数据库操作、API 调用 |
| **Resources（资源）** | 可读取的数据源 | 文件、文档、配置 |
| **Prompts（提示）** | 可复用的提示模板 | 代码审查提示、摘要模板 |

**采用率：** Tools（99%）、Resources（30-35%）、Prompts（约 30%）

### MCP 的优势

1. **一次编写，到处使用。** 写一个 MCP 服务器，所有 MCP 客户端都能用
2. **动态发现。** 智能体自动发现和列出可用工具
3. **生态杠杆。** 利用社区已有的 MCP 服务器（GitHub、Slack、数据库等）
4. **关注点分离。** 工具实现和智能体逻辑解耦

### ADK 中使用 MCP

```python
from adk.tools import MCPToolset

# 连接到 MCP 服务器
mcp_tools = MCPToolset(
    server_url="https://mcp.example.com/server"
)

# 智能体自动获得这些工具
agent.tools = [mcp_tools]
```

ADK 有 60+ 预构建 MCP 集成。

### MCP 的局限性

| 问题 | 描述 | 缓解 |
|------|------|------|
| **工具影子** | MCP 工具与内置工具功能重叠 | 审查工具集，避免冗余 |
| **上下文窗口膨胀** | 太多工具描述占满上下文 | 精简工具数量，动态加载 |
| **无原生范围限制** | MCP 本身不限制工具权限 | 在服务器端加权限控制 |
| **信任/供应链** | 第三方 MCP 服务器可能不可信 | 审查代码，限制权限 |
| **延迟** | 网络调用增加响应时间 | 本地工具优先，缓存结果 |

### MCP vs. CLI 工具

| 方面 | CLI 工具 | MCP 服务器 |
|------|----------|-----------|
| Token 消耗 | 低（约 1,400 token） | 高（约 44,000 token，32 倍） |
| 可靠性 | 100%（确定性） | 72%（可能失败） |
| 安全/认证 | 有限 | OAuth 2.1 with PKCE |
| 多用户环境 | 不适合 | 适合 |
| 工具发现 | 手动 | 自动 |
| 结构化 I/O | 需要额外处理 | 内建支持 |

**选择指南：**
- 简单脚本、单次使用 → CLI
- 需要认证、多用户、工具发现 → MCP

## A2A：智能体到智能体协议

### 核心概念

A2A 让智能体之间以标准化方式通信：

```
[客户端智能体]                    [服务端智能体]
     │                                │
     │── 1. 查询 Agent Card ────────→│
     │←── Agent Card（JSON）─────────│
     │                                │
     │── 2. 创建 Task ───────────────→│
     │←── Task ID ───────────────────│
     │                                │
     │── 3. 轮询/监听 Events ────────→│
     │←── Event Stream（SSE）────────│
     │                                │
     │←── 4. Artifact（结果）────────│
```

### Agent Card

每个 A2A 智能体都有一个 Agent Card——描述它能做什么的 JSON：

```json
{
  "name": "research-agent",
  "description": "专业研究智能体，擅长信息检索和综合分析",
  "version": "1.0.0",
  "capabilities": ["research", "summarization", "fact-checking"],
  "endpoints": {
    "task": "https://agents.example.com/research/tasks",
    "events": "https://agents.example.com/research/events"
  }
}
```

### Tasks 和 Artifacts

- **Task**：一个工作单元，有输入、状态和进度
- **Artifact**：任务的输出结果（文档、数据、代码等）

### 事件队列（Event Queues）

A2A 使用 Server-Sent Events（SSE）流式推送任务进展：

```json
{"event": "task_started", "task_id": "abc123"}
{"event": "progress", "task_id": "abc123", "message": "搜索中..."}
{"event": "progress", "task_id": "abc123", "message": "分析结果..."}
{"event": "task_completed", "task_id": "abc123", "artifact": {...}}
```

### A2A vs. 直接 API 调用

| 方面 | A2A | 直接 API |
|------|-----|----------|
| **交互模式** | 面向任务，异步 | 面向请求，同步 |
| **状态管理** | 服务端跟踪进度 | 无状态 |
| **进度反馈** | 事件流实时推送 | 需要轮询 |
| **发现** | Agent Card 自动发现 | 需要文档 |
| **最适合** | 智能体间协作 | 简单服务调用 |

---

## MCP vs. A2A 对比

| 方面 | MCP | A2A |
|------|-----|-----|
| **层级** | 工具层 | 智能体层 |
| **类比** | 电源适配器 | 电话通话 |
| **通信对象** | 智能体 ↔ 工具 | 智能体 ↔ 智能体 |
| **交互模式** | 函数调用 | 面向任务 |
| **通信内容** | 参数 → 返回值 | 目标 → 成果 |
| **典型用途** | 调用 API、查询数据库 | 委派研究任务、协调工作 |

### 形象类比

- **MCP** = 万能电源适配器。不管什么电器，插上就能用。不管什么工具，连上 MCP 就能被智能体调用。

- **A2A** = 打电话。你知道对方号码，拨过去，说事，挂断。不需要知道对方用什么设备。

## 分层架构：MCP + A2A 协同

实际系统中，两个协议通常一起使用：

```
┌──────────────────────────────────────┐
│            编排层                     │
│   协调智能体（决定做什么）             │
├──────────────────────────────────────┤
│            A2A 层                     │
│   智能体间通信（委派任务、接收结果）   │
├──────────────────────────────────────┤
│      MCP 层       │      MCP 层      │
│   工具服务器 A     │    工具服务器 B   │
│   （搜索、DB）     │   （代码、API）   │
└──────────────────────────────────────┘
```

### 具体场景：旅行规划

```
用户："帮我规划下周去东京的旅行"

编排智能体（协调者）
  │
  ├── A2A → 研究智能体
  │     ├── MCP → 搜索工具（查景点、餐厅）
  │     └── MCP → 天气工具（查天气预报）
  │
  ├── A2A → 预订智能体
  │     ├── MCP → 航班 API
  │     └── MCP → 酒店 API
  │
  └── A2A → 预算智能体
        ├── 综合所有信息
        └── 生成预算方案

编排智能体整合所有结果 → 完整行程计划
```

## 安全考量

### MCP 安全

- **认证**：使用 OAuth 2.1 with PKCE 保护 MCP 服务器
- **授权**：限制哪些智能体可以访问哪些工具
- **输入验证**：在 MCP 服务器端验证所有输入
- **审计日志**：记录所有工具调用

### A2A 安全

- **Agent Card 验证**：确认通信对方是合法的
- **任务隔离**：每个任务独立运行，不共享状态
- **权限边界**：限制子智能体可调用的工具范围
- **结果验证**：验证收到的 Artifact 是否符合预期

### Google Cloud 安全服务

- **Apigee** —— 作为 MCP 网关，管理 API 安全
- **Model Armor** —— 输入输出内容安全过滤
- **Cloud IAM** —— 身份和访问管理

---

## 核心要点

1. **MCP 解决智能体-工具通信。** 标准化工具接口，避免 N×M 集成负担。

2. **A2A 解决智能体-智能体通信。** 标准化任务委派和结果返回。

3. **两者互补而非竞争。** 分层架构中，MCP 在底层连接工具，A2A 在中间层连接智能体。

4. **权衡无处不在。** MCP 带来灵活性的同时增加 token 消耗和延迟。评估后再决定用 MCP 还是 CLI。

5. **安全需要纵深防御。** 从认证、授权、输入验证到审计日志，每层都不能少。
