## ELI5：设计模式就像食谱

想象你要做晚饭。你可以随手抓食材，希望结果不错。或者你可以按照食谱来——这是已被验证有效的一组步骤。

设计模式就是构建 AI 智能体的食谱。它们是经过实践检验的、组织智能体如何思考、行动和学习的方式。就像菜谱书有不同菜肴的不同食谱，不同类型的智能体行为有不同的模式。

就像优秀的大厨可能融合多个食谱中的技法，最好的智能体通常将多种模式结合在一起。

## 为什么设计模式重要

如果你编程有一段时间了，可能熟悉观察者、策略或工厂等设计模式。这些模式给工程师提供了共同的词汇和解决常见问题的成熟方案。

智能体设计模式服务于同样的目的，但针对 AI 智能体。它们描述了智能体在以下方面的重复性结构：

- 如何**推理**问题
- 如何在世界中**采取行动**
- 如何从结果中**学习**
- 如何**改进**自身的输出

没有这些模式，构建智能体就像写意大利面条代码——一切相互缠绕，难以调试。有了这些模式，你就拥有了更易于构建、测试和维护的清晰架构。

### 从简单到智能体化

并非每个 LLM 交互都需要设计模式。以下是一个大致的谱系：

| 级别 | 描述 | 示例 | 所需模式 |
|-------|------------|---------|----------------|
| **简单提示** | 一个问题，一个答案 | "法国的首都是哪里？" | 无 |
| **结构化输出** | LLM 格式化其响应 | "以 JSON 返回" | 无 |
| **链式** | 多个 LLM 调用的序列 | 摘要，然后翻译 | 最少 |
| **智能体** | LLM 决定下一步做什么 | 研究一个主题并撰写报告 | ReAct、工具使用、规划 |
| **多智能体** | 多个智能体协作 | 构建软件的智能体团队 | 以上所有 + 协调 |

一旦你从简单链式进入真正的智能体行为——LLM 在做下一步决策——设计模式就变得重要了。

## 模式一：ReAct（推理 + 行动）

ReAct 是最基础的智能体设计模式。如果你只学一种模式，就学这个。

### 核心思想

ReAct 将**推理**（思考该做什么）与**行动**（实际去做）交织在一个循环中。智能体：

1. **思考**当前情况
2. **行动**，调用工具或采取步骤
3. **观察**结果
4. **重复**直到任务完成

### 餐厅类比

想想餐厅里的服务员：

1. **思考：** "5 号桌点了意面。我需要确认厨房是否做好了。"
2. **行动：** 走进厨房询问。
3. **观察：** 厨师说还需要 5 分钟。
4. **思考：** "好的，我可以先去确认 3 号桌的饮料。"
5. **行动：** 去 3 号桌补充饮料。
6. **观察：** 3 号桌很满意。
7. **思考：** "时间差不多了，再去看看 5 号桌的意面。"
8. **行动：** 回厨房。
9. **观察：** 意面做好了。
10. **行动：** 将意面送到 5 号桌。

服务员不会提前规划整个班次。他们思考、行动、观察、适应——持续循环。

### ReAct 的样子

以下是一般流程：

```
用户："东京的天气怎么样，我需要带雨伞吗？"

智能体思考：我需要查找东京当前的天气。我将使用天气工具。
智能体行动：调用 weather_api(location="Tokyo")
观察：{"temp": 18, "condition": "rain", "forecast": "明天仍有雨"}

智能体思考：东京现在在下雨，预计雨天持续到明天。
            用户应该带雨伞。
智能体行动：回复用户。

智能体回复："东京目前气温 18 摄氏度并在下雨，预计雨天持续到明天。
             一定要带雨伞！"
```

### ReAct 的关键特征

- **推理和行动交织。** 智能体不是盲目行动——它在每个步骤前都会思考。
- **以观察为基础。** 每个想法都基于智能体实际观察到的内容，而不只是假设。
- **灵活。** 智能体可以根据发现的内容改变方向。
- **透明。** 推理轨迹使调试智能体的思考过程更容易。

### 何时使用 ReAct

| 适合 | 不适合 |
|----------|----------|
| 需要外部信息的任务 | 纯文本生成任务 |
| 路径不确定的多步骤问题 | 简单的问答 |
| 需要审计记录的情况 | 对延迟敏感的应用 |
| 需要根据新信息调整的任务 | 有固定已知序列的任务 |

### 常见陷阱

- **推理循环。** 智能体反复思考同一个想法而没有进展。添加最大迭代计数。
- **幻觉行动。** 智能体"调用"一个不存在的工具。在执行前验证工具名称。
- **忽视观察。** 智能体忽略工具返回的内容，继续基于之前的假设行动。确保观察清晰地注入上下文。

## 模式二：反思

### 核心思想

在反思模式中，智能体审查自己的输出并改进它。不是产生一个响应就结束，而是生成草稿、批评它，然后修订。

### 写作者类比

想想一个正在写文章的作者：

1. **起草：** 写第一个版本。
2. **审查：** 再读一遍。"嗯，引言太弱了，第 3 段与第 1 段矛盾。"
3. **修订：** 重写引言并修正矛盾。
4. **再次审查：** "好多了。但结论需要更有力的行动号召。"
5. **再次修订：** 改进结论。
6. **完成：** 最终版本比初稿强多了。

没有经验的作者会直接发出初稿。同样，能反思输出的智能体会产生显著更好的结果。

### 反思的样子

```
第 1 步 - 生成：
  智能体对用户的请求产生初始响应。

第 2 步 - 批评：
  智能体（或单独的评论者）审查响应：
  "这段代码在第 12 行有个 Bug——循环索引差了一个。
   此外，函数缺少对空输入的错误处理。"

第 3 步 - 修订：
  智能体修复发现的问题，产生改进版本。

第 4 步 - 评估：
  "Bug 已修复，错误处理已添加。代码现在处理了边缘情况。
   满足要求。"
```

### 反思的变体

| 变体 | 工作原理 | 示例 |
|-----------|-------------|---------|
| **自我反思** | 同一个 LLM 审查自己的输出 | "检查你的代码是否有 Bug" |
| **评论者智能体** | 单独的 LLM 实例审查 | 专门的代码审查者智能体 |
| **基于评分标准** | 反思由特定标准指导 | "检查：准确性、完整性、语气" |
| **测试驱动** | 输出通过具体检查进行测试 | 运行单元测试，检查格式 |

### 何时使用反思

| 适合 | 不适合 |
|----------|----------|
| 代码生成（在发布前捕获 Bug） | 实时对话响应 |
| 写作任务（提高清晰度和结构） | 简单事实查询 |
| 复杂推理（捕获逻辑错误） | 速度比质量更重要的任务 |
| 质量比速度更重要的任何任务 | 有客观可验证答案的任务 |

### 实践建议

- **设置最大反思轮次。** 两到三轮通常就足够了。超过这个数通常收益递减。
- **使用具体标准。** "使它更好"太模糊。"检查 SQL 注入漏洞、缺失的错误处理和性能问题"给了反思明确的焦点。
- **考虑为评论者使用不同的模型或提示。** 全新的视角比用同样的视角重新阅读更能发现问题。
- **追踪变化了什么。** 记录每次修订之间的差异，以验证反思确实改进了内容。

## 模式三：工具使用

### 核心思想

在工具使用模式中，智能体决定调用哪些工具、使用什么参数、以什么顺序。LLM 作为编排外部能力的推理引擎，而不是试图自己做所有事情。

我们在[第三课](../03-tools-giving-agents-hands/README_ZH.md)深入介绍了工具。本节聚焦于智能体如何决定使用工具这一*模式*。

### 工匠类比

想想一个有经验的木匠在做项目。他有满满一个工作室的工具——锯子、钻头、砂纸机、夹具。对于项目的每个步骤，他：

1. 评估下一步需要做什么
2. 为这个具体步骤选择正确的工具
3. 使用工具
4. 检查结果
5. 决定下一步

他不会什么都用锤子。他选择适合工作的工具。有时他意识到项目中间需要一个他没想到会用的工具。

### 工具使用的样子

```
用户："找出 GitHub 上关于 AI 智能体的前 3 个热门仓库，
      并总结每个仓库的用途。"

智能体思考：我需要在 GitHub 上搜索热门的 AI 智能体仓库。
智能体行动：github_search(query="AI agents", sort="stars", limit=3)
观察：[repo1: "autogen", repo2: "crewai", repo3: "langchain-agents"]

智能体思考：现在我需要获取每个仓库的详细信息。
智能体行动：github_get_repo(name="autogen")
观察：{description: "...", readme: "...", stars: 42000}

智能体行动：github_get_repo(name="crewai")
观察：{description: "...", readme: "...", stars: 28000}

智能体行动：github_get_repo(name="langchain-agents")
观察：{description: "...", readme: "...", stars: 19000}

智能体思考：我现在有了足够的信息来总结所有三个仓库。
智能体回复："以下是热门 AI 智能体仓库前三名..."
```

### 工具选择策略

智能体使用几种策略来决定使用哪个工具：

| 策略 | 描述 | 权衡 |
|----------|------------|-----------|
| **直接匹配** | 工具名称/描述与需求匹配 | 快速，但如果工具命名不佳则脆弱 |
| **能力推理** | 智能体推理每个工具能做什么 | 更灵活，但使用更多 token |
| **少样本示例** | 提示中的示例展示何时使用每个工具 | 可靠，但占用上下文空间 |
| **工具推荐** | 系统根据查询建议相关工具 | 减少 LLM 的决策负担 |

### 并行 vs. 顺序工具调用

有些任务允许同时调用多个工具：

- **顺序：** 搜索用户，然后查询其订单历史（需要先获取用户 ID）
- **并行：** 检查三个不同城市的天气（都是独立的）

并行工具调用可以显著减少延迟。在设计智能体时，识别哪些工具调用是独立的，可以同时运行。

### 何时使用工具使用

这种模式适用于几乎任何与外部系统交互的智能体。关键设计决策是：

- **多少个工具？** 从少量开始。有 3-5 个精心设计工具的智能体通常优于有 50 个设计糟糕工具的智能体。
- **工具模式的描述性如何？** 更好的描述带来更好的工具选择。
- **工具失败时怎么办？** 好的智能体能优雅地处理错误——重试、尝试替代工具或向用户寻求帮助。

## 模式四：规划

### 核心思想

在规划模式中，智能体在执行之前先创建一个计划。不是像 ReAct 那样逐步决策，而是提前思考并制定结构化方案。

### 项目经理类比

想象一位项目经理收到构建新功能的请求：

1. **分解：** "我们需要更新数据库模式、编写 API 端点、构建 UI 并添加测试。"
2. **排序工作：** "先做模式，然后 API，然后 UI，然后测试——每个都依赖前一个步骤。"
3. **分配资源：** "数据库工作交给后端团队，UI 交给前端团队。"
4. **执行和跟踪：** 按计划推进，逐项完成。
5. **必要时调整：** "模式变更比预期复杂——让我重新规划时间线。"

### 规划的样子

```
用户："写一篇关于 Kubernetes 安全最佳实践的全面博客文章。"

智能体计划：
  1. 研究当前的 Kubernetes 安全威胁和 CVE
  2. 确定 5-7 个最重要的安全最佳实践
  3. 为每个实践找具体示例和命令
  4. 写出包含引言、主要章节和结论的提纲
  5. 起草每个章节
  6. 审查全文的准确性和流畅性
  7. 添加代码示例和格式

智能体执行：
  [按顺序执行第 1-7 步，按需调整]
```

### 规划策略

| 策略 | 工作原理 | 最适合 |
|----------|-------------|----------|
| **顺序规划** | 创建线性步骤列表 | 简单、易理解的任务 |
| **层次规划** | 分解为高层目标，然后是子任务 | 复杂的多阶段项目 |
| **条件规划** | 在计划中包含 if/then 分支 | 结果不确定的任务 |
| **迭代规划** | 规划几步，执行，重新规划 | 后续步骤依赖早期结果的任务 |

### 先规划再执行 vs. ReAct

这两种模式代表了不同的哲学：

| 方面 | 规划 | ReAct |
|--------|----------|-------|
| **何时做决策** | 大部分在前期 | 逐步进行 |
| **适应性** | 需要明确的重新规划 | 自然地适应 |
| **效率** | 可以并行化独立步骤 | 通常是顺序的 |
| **透明度** | 全部计划在前期可见 | 每步推理可见 |
| **浪费工作的风险** | 如果计划错误则较高 | 较低，随时适应 |
| **最适合** | 结构良好的任务 | 探索性任务 |

实际上，大多数智能体混合两种方法：前期做粗略的计划，然后在执行时使用 ReAct 风格的推理。

### 何时使用规划

| 适合 | 不适合 |
|----------|----------|
| 结构清晰的多步骤任务 | 简单的单步任务 |
| 顺序很重要的任务 | 纯响应式/对话型智能体 |
| 可以并行化的工作 | 路径完全未知的任务 |
| 需要进度跟踪的项目 | 快速临时的请求 |

## 比较这些模式

以下是帮助你选择的并排比较：

| 模式 | 核心思想 | 优势 | 弱点 | 成本 |
|---------|-----------|----------|----------|------|
| **ReAct** | 思考-行动-观察循环 | 灵活、透明 | 可能较慢，可能循环 | 中等（多次 LLM 调用） |
| **反思** | 自我审查和改进 | 更高质量输出 | 增加延迟 | 高（多次通过） |
| **工具使用** | 编排外部工具 | 扩展智能体能力 | 依赖工具质量 | 可变（取决于工具） |
| **规划** | 先规划再执行 | 结构化、高效 | 计划错误时脆弱 | 中到高（规划 + 执行） |

### 决策流程图

问自己以下问题：

1. **智能体需要外部信息或行动吗？** 是 -> 工具使用
2. **任务是多步骤且路径不确定的吗？** 是 -> ReAct
3. **质量至关重要且任务有明确标准吗？** 是 -> 反思
4. **任务复杂但结构良好吗？** 是 -> 规划
5. **大多数答案都是"是"吗？** -> 组合模式

## 组合模式

真实世界的智能体几乎从不单独使用一种模式。最有效的智能体将模式层叠在一起。

### 常见组合

**ReAct + 工具使用**（最常见的组合）

智能体推理该做什么，使用工具采取行动，观察结果，再次推理。这是大多数实际智能体的主干。

```
思考 -> 使用工具 -> 观察 -> 思考 -> 使用工具 -> 观察 -> 响应
```

**规划 + ReAct + 工具使用**

智能体创建计划，然后使用 ReAct 风格的推理和工具执行每个步骤。

```
规划 -> [思考 -> 行动 -> 观察] -> [思考 -> 行动 -> 观察] -> ... -> 完成
```

**规划 + 反思**

智能体创建计划，执行它，然后在交付前审查整体输出。

```
规划 -> 执行 -> 反思 -> 修订 -> 交付
```

**全套：规划 + ReAct + 工具使用 + 反思**

对于复杂、高风险的任务，你可能会使用所有四种：

```
规划方法
  -> 用 ReAct + 工具执行每个步骤
    -> 对整体结果反思
      -> 必要时修订
        -> 交付
```

### 示例：代码生成智能体

代码生成智能体如何组合这些模式：

1. **规划：** "我需要写一个 REST API。步骤：定义数据模型，创建端点，添加验证，写测试。"
2. **ReAct + 工具使用：** 对每个步骤，智能体推理该做什么，使用工具（文件读取器、代码搜索、代码检查工具）收集信息并编写代码。
3. **反思：** 写完代码后，智能体对照最佳实践审查它。"这处理了错误吗？输入是否经过验证？是否有安全问题？"
4. **修订：** 智能体修复反思中发现的问题。

### 不应该组合的情况

更多模式并不总是更好。每种模式增加了：

- **延迟：** 更多 LLM 调用意味着更多时间
- **成本：** 更多 token 意味着更多钱
- **复杂性：** 更多活动部件意味着更多调试

对于一个简单的问答智能体，ReAct + 工具使用可能就是你所需要的全部。将全套保留给质量足以证明成本合理的复杂高价值任务。

## 核心要点

1. **智能体设计模式是经过验证的蓝图**，用于组织智能体的思考和行动方式。它们给你提供共同的词汇和架构的起点。

2. **ReAct 是基础。** 思考-行动-观察循环是最基本的模式，也是大多数智能体的起点。

3. **反思大幅提高质量**，但需要时间和 token。在质量比速度更重要时使用它。

4. **工具使用超越了 LLM 内置知识的范畴。** 好的工具设计与好的提示设计同等重要。

5. **规划为复杂任务带来结构。** 当任务有充分理解且步骤可以提前安排时效果最好。

6. **谨慎地组合模式。** 更多模式意味着更多能力，但也意味着更多复杂性和成本。从简单开始，按需添加模式。

7. **没有单一最佳模式。** 正确的选择取决于你的任务、质量要求以及延迟和成本预算。
