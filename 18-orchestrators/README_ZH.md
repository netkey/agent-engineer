# 第十八课：编排器——管理智能体控制流

## 简介

工具给智能体双手，技能给智能体知识，但谁来指挥这一切？

编排器是智能体系统的"导演"。它决定：
- 智能体按什么顺序执行
- 何时并行、何时串行
- 出错时怎么处理
- 上下文如何组装和传递

本课介绍六种核心编排模式、ADK 中的实现、框架对比和最佳实践。

---

## ELI5：电影导演

一部电影有导演、摄影师、灯光师、音效师。

导演不拿摄像机，不布光，不录声音——他决定谁在什么时候做什么。

编排器就是智能体系统的导演。它不执行具体任务，它**指挥其他智能体什么时候执行什么**。

---

## 编排器的四个核心关注点

1. **控制流** —— 智能体按什么顺序、什么条件执行
2. **上下文组装** —— 每个智能体看到什么信息
3. **状态管理** —— 跟踪整体进度和中间结果
4. **错误处理** —— 某步失败时怎么办

---

## 核心智能体循环

```
接收输入
  → 组装上下文（系统指令 + 历史 + 工具定义）
  → LLM 推理（思考下一步）
  → 决定行动（工具调用或子智能体委派）
  → 执行行动
  → 观察结果
  → 决定：完成？→ 返回输出
         未完成？→ 循环
```

编排器管理这个循环的节奏和边界。

---

## 确定性 vs. 动态编排

| 方面 | 确定性 | 动态 |
|------|--------|------|
| **控制流** | 硬编码，按预设顺序 | LLM 决定下一步 |
| **可预测性** | 高——始终走同一条路 | 低——每次可能不同 |
| **速度** | 快——无额外推理 | 较慢——需要 LLM 决策 |
| **灵活性** | 低——路径固定 | 高——根据情况调整 |
| **最适合** | 已知流程、合规要求 | 探索性任务 |

### 混合方法

实际中最常用混合方式：确定性的骨架 + 动态的细节。

```
确定性的阶段：[研究 → 写作 → 审查]
  研究阶段内：动态探索（不知道会找到什么）
  写作阶段内：动态生成（但知道最终需要一篇文）
  审查阶段内：动态评估（但知道审查标准）
```

## 六种核心编排模式

### 1. 顺序（流水线）

智能体按固定顺序执行，每个接收上一个的输出。

```python
from adk.agents import SequentialAgent

pipeline = SequentialAgent(
    name="content_pipeline",
    sub_agents=[researcher, writer, editor]
)
```

**最适合：** 有明确阶段顺序的任务
**示例：** 内容生成（研究→写作→编辑）、数据处理（采集→清洗→分析）

### 2. 并行（扇出/汇聚）

多个智能体同时执行独立任务，然后汇聚结果。

```python
from adk.agents import ParallelAgent

reviewer = ParallelAgent(
    name="code_reviewer",
    sub_agents=[python_reviewer, js_reviewer, security_reviewer]
)
```

**最适合：** 独立任务可以并行处理
**示例：** 多维度代码审查、多个数据源同时查询

### 3. 循环（迭代）

智能体重复执行直到满足退出条件。

```python
from adk.agents import LoopAgent

iterative_writer = LoopAgent(
    name="iterative_writer",
    sub_agents=[writer, critic],
    max_iterations=3  # 必须有最大迭代次数
)
```

**最适合：** 需要逐步改进的任务
**示例：** 写作+审查循环、搜索+精炼循环
**模式：** 生成者-评论者（Generator-Critic）

### 4. 路由（交接）

根据输入类型将请求路由到不同的专业智能体。

```
用户请求 → [路由智能体]
              │
              ├── 技术问题 → 技术支持智能体
              ├── 账单问题 → 财务智能体
              └── 账号问题 → 安全智能体
```

**最适合：** 多领域分类
**示例：** 客服路由、意图分类

### 5. 层级（协调者-工作者）

一个协调智能体将任务分解并分配给工作者智能体。

```python
from adk.tools import AgentTool

coordinator = LlmAgent(
    name="coordinator",
    instruction="将任务分解并分配给合适的子智能体。"
)

# 将子智能体封装为工具
research_tool = AgentTool(research_agent)
writing_tool = AgentTool(writing_agent)

coordinator.tools = [research_tool, writing_tool]
```

**最适合：** 复杂任务需要中央协调
**示例：** 项目管理、多步骤调查

### 6. 群聊（圆桌）

多个智能体平等参与讨论，共同达成共识。

**最适合：** 需要共识的决策
**示例：** 架构评审、风险评估
**建议：** 不超过 3 个智能体，超过容易混乱

### 模式对比

| 模式 | 可预测性 | 灵活性 | Token 成本 | 最适合 |
|------|----------|--------|-----------|--------|
| 顺序 | 高 | 低 | 低 | 有明确阶段的任务 |
| 并行 | 高 | 中 | 中 | 独立任务 |
| 循环 | 中 | 高 | 高 | 迭代优化 |
| 路由 | 高 | 中 | 低 | 多领域分类 |
| 层级 | 中 | 高 | 高 | 复杂分解 |
| 群聊 | 低 | 高 | 高 | 共识决策 |

## 决策流程图

```
任务需要多步骤吗？
  │
  +-- 否 → 单个 LlmAgent 就够了
  │
  +-- 是 → 步骤有固定顺序吗？
  │         │
  │         +-- 是 → 步骤可以并行吗？
  │         │         │
  │         │         +-- 是 → ParallelAgent
  │         │         +-- 否 → SequentialAgent
  │         │
  │         +-- 否 → 需要迭代改进吗？
  │                   │
  │                   +-- 是 → LoopAgent
  │                   +-- 否 → 层级（AgentTool 分解）
```

---

## 组合模式

实际中经常组合使用多种模式：

```python
# 顺序管道包含并行和循环
pipeline = SequentialAgent(
    name="full_pipeline",
    sub_agents=[
        research_agent,       # 单个智能体
        ParallelAgent(        # 并行审查
            sub_agents=[security_reviewer, performance_reviewer]
        ),
        LlmAgent(...),        # 综合结果
        LoopAgent(            # 迭代优化
            sub_agents=[writer, critic],
            max_iterations=3
        )
    ]
)
```

## 最佳实践

### 1. 从简单开始

```
层级 1：单个 LlmAgent
层级 2：SequentialAgent（线性流水线）
层级 3：ParallelAgent + LoopAgent
层级 4：多层嵌套 + 自定义逻辑
```

不要一上来就做多层层嵌套。大多数需求用层级 1-2 就够了。

### 2. 模型匹配任务

| 任务 | 模型 |
|------|------|
| 编排逻辑（协调者） | Pro（需要推理） |
| 具体执行（工作者） | Flash（需要速度） |
| 审查评估 | Pro（需要判断力） |

### 3. 设置迭代限制

所有循环智能体必须设 `max_iterations`。建议 3-5 次。

### 4. 步骤间验证

```python
def validate_step_result(result):
    if result.quality_score < threshold:
        raise ValidationError("结果质量不足，需要重试或人工介入")
```

### 5. 管理上下文

每经过一个智能体，上下文都在增长。使用摘要和截断防止上下文耗尽。

### 6. 可观测性

记录每个智能体的输入、输出、耗时和状态。调试多智能体系统时这是最有价值的信息。

### 7. 为失败设计

- 每个步骤有超时
- 关键步骤有重试
- 有降级路径（部分功能不可用时怎么办）
- 最终有优雅的错误提示

## 反模式

| 反模式 | 问题 | 修复 |
|--------|------|------|
| **编排过度** | 简单的任务用了复杂的编排 | 从单智能体开始，按需添加 |
| **无专业化** | 子智能体和父智能体做一样的事 | 每个子智能体有明确的专长 |
| **共享可变状态** | 多个智能体修改同一状态 | 不可变数据流或锁机制 |
| **无迭代限制** | 循环智能体可能无限循环 | 设置 `max_iterations` |
| **上下文膨胀** | 太多中间结果填满窗口 | 摘要、截断、外部化 |
| **错误编排类型** | 需要灵活性的地方用了确定性 | 根据任务选择编排类型 |

---

## 核心要点

1. **编排器管理控制流。** 它决定谁在什么时候执行什么。

2. **六种核心模式：** 顺序、并行、循环、路由、层级、群聊。大多数系统用前四种就够了。

3. **混合方法最实用。** 确定性骨架 + 动态细节。

4. **从简单开始。** 单智能体 → 顺序 → 并行/循环 → 层级。不要过度编排。

5. **模型匹配任务。** 协调者用强推理模型，工作者用快速模型。

6. **必须有安全边界。** 迭代限制、超时、验证、降级路径缺一不可。

7. **可观测性是调试的命脉。** 没有每个智能体的输入输出日志，出了问题无从查起。
