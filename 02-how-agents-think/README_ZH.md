# 第二课：智能体如何思考——LLM 作为推理引擎

## 简介

在第一课中，我们说智能体由三个部分组成：模型（大脑）、工具（双手）和编排层（控制循环）。本课聚焦于大脑。

语言模型是智能体最重要的组件。它负责读取用户目标、推理应该做什么、决定调用哪些工具、解读结果并生成最终答案。智能体系统中的其他一切都是为了支持或扩展模型的能力。

理解 LLM 的工作原理——哪怕只是高层次的了解——会让你成为更好的智能体构建者。你会明白为什么有些提示有效而另一些无效，理解智能体有时会出轨的原因，并能在模型选择、提示设计和系统架构方面做出更明智的决策。

---

## LLM：智能体的推理引擎

大型语言模型是在海量文本数据上训练的神经网络。其核心只做一件事：**预测下一个 token**（大致相当于下一个词或词的片段），基于之前出现的所有内容。

这听起来简单，但训练产生的涌现能力令人瞩目：

- **理解**：理解复杂问题和指令
- **推理**：处理多步骤逻辑问题
- **规划**：将目标分解为子任务
- **代码生成**：编写和调试软件
- **工具选择**：决定调用哪个函数以及使用什么参数
- **摘要**：将长文档提炼为要点
- **翻译**：在语言、格式和表示之间转换

### LLM 擅长的事情

| 能力 | 智能体场景示例 |
|---|---|
| 自然语言理解 | 解析用户请求："取消我最近的订单" |
| 推理和规划 | 决策："首先需要找到订单，然后检查是否可取消，再执行取消" |
| 工具选择 | 选择调用 `lookup_order(user_id, sort="recent")` |
| 输出格式化 | 返回干净的 JSON 响应或友好的消息 |
| 错误解读 | 读取 API 错误并决定用不同参数重试 |
| 上下文综合 | 将多次工具调用的结果组合成连贯的答案 |

### LLM 做不到的事情（如果没有帮助）

| 局限性 | 为何重要 |
|---|---|
| 无实时数据访问 | 模型的知识有训练截止日期 |
| 无计算保证 | LLM 可能算错——它们预测 token，不进行计算 |
| 无持久记忆 | 每次对话都从头开始，除非你构建了记忆功能 |
| 无行动能力 | 没有工具，模型只能生成文本 |
| 幻觉风险 | 模型可能自信地生成听起来合理但不正确的信息 |
| 上下文窗口限制 | 模型一次能处理的文本量有上限 |

这就是智能体存在的原因。工具弥补了模型无法行动和访问实时数据的不足；编排层弥补了其缺乏持久记忆和容易偏轨的问题。

## 语言模型处理信息的方式

你不需要深入理解 Transformer 架构就能构建智能体，但理解三个关键概念会帮助你写出更好的提示，设计更好的系统。

### Token：语言的基本单位

LLM 不读取字符或词语，而是读取 **token**——模型在训练时学会识别的文本片段。一个 token 可能是一个完整的词、词的一部分，或者一个标点符号。

**示例：**

| 文本 | 大约的 Token 数 |
|---|---|
| "Hello" | 1 个 |
| "Hello, world!" | 3 个 |
| "ChatGPT is amazing" | 4 个 |
| 一个典型代码函数（20 行） | 100-300 个 |
| 一整页英文文本 | 约 500-700 个 |

**为什么这对智能体很重要：**

- **计费**：大多数 API 按 token（输入 + 输出）收费。智能体工作流比单次提示使用多得多的 token，因为循环的每次迭代都要发送完整上下文。
- **速度**：token 越多 = 生成时间越长。保持工具描述简洁。
- **上下文限制**：模型单次调用能处理的 token 数量有上限。如果智能体积累的上下文超过这个上限，就会丢失信息。

### 上下文窗口：模型的工作记忆

**上下文窗口**是 LLM 在单次请求中能处理的总 token 数量。把它想象成模型的桌子——它需要参考的所有内容都必须放在这张桌子上。

| 模型 | 上下文窗口 |
|---|---|
| Gemini 2.5 Pro | 1,000,000 token |
| Gemini 2.0 Flash | 1,000,000 token |
| GPT-4o | 128,000 token |
| Claude 3.5 Sonnet | 200,000 token |

**智能体调用时，上下文窗口中包含的内容：**

```
+------------------------------------------+
| 系统指令（"你是一个..."）               | 约 200-500 token
+------------------------------------------+
| 工具定义（名称、描述、                  | 约 500-2000 token
| 参数模式）                              |
+------------------------------------------+
| 对话历史                                | 可变
+------------------------------------------+
| 之前的工具调用和结果                    | 可变（增长很快）
+------------------------------------------+
| 当前用户消息                            | 可变
+------------------------------------------+
| = 总量必须在上下文窗口内               |
+------------------------------------------+
```

**为什么这对智能体很重要：**

随着智能体执行多个步骤，每次工具调用和返回结果都会扩大上下文。一个 5 步智能体工作流可能积累数千 token 的工具结果。如果不加注意，可能在任务中途耗尽上下文窗口。

应对策略：
- **总结**中间结果，而不是保留原始数据
- **截断**长工具输出，只保留相关部分
- 为复杂的多步骤任务**使用大上下文窗口的模型**
- 实现**滑动窗口**，丢弃较旧、相关性较低的上下文

### 注意力机制：模型的聚焦方式

**注意力机制**使模型能够判断上下文的哪些部分与当前决策相关。在决定下一个要生成的 token 时，模型对输入的不同部分赋予不同的权重。

把它想象成阅读一篇长文档并给重要内容划重点。模型会"高亮"与当前任务最相关的 token。

**为什么这对智能体很重要：**

- **将重要信息放在模型容易找到的位置。** 模型倾向于更多关注上下文的开头和结尾。关键指令应该放在系统提示中（开头）或靠近用户问题处（结尾）。
- **具体清晰。** 模糊的指令迫使模型猜测什么重要。具体的指令让注意力机制更容易抓住正确信息。
- **结构有帮助。** 清晰的标题、编号列表和一致的格式帮助模型解析并关注正确的内容。

## 推理策略

智能体如何"思考"问题，在很大程度上取决于你如何提示模型。不同的推理策略产生迥异的结果，对复杂任务尤为明显。

### 思维链（Chain-of-Thought，CoT）

**是什么：** 提示模型在给出最终答案之前，逐步思考问题。

**工作原理：** 与其直接跳到答案，模型生成中间推理步骤。这大幅提升了需要逻辑、数学或多步骤分析的任务的准确性。

**不使用 CoT 的示例：**
```
提示："一台服务器处理 100 请求/秒，我们有 3 台服务器，
      其中 40% 的流量流向服务器 1，服务器 1 处理多少请求/秒？"

模型回复："120 请求/秒"（错误）
```

**使用 CoT 的示例：**
```
提示："逐步思考。一台服务器处理 100 请求/秒，我们有 3 台服务器，
      其中 40% 的流量流向服务器 1，服务器 1 处理多少请求/秒？"

模型回复：
"第 1 步：总容量为 3 台 × 100 请求/秒 = 300 请求/秒
 第 2 步：服务器 1 接收 40% 的总流量
 第 3 步：300 的 40% = 120 请求/秒
 第 4 步：服务器 1 能处理 100 请求/秒，但接收 120 请求/秒
 答案：服务器 1 接收 120 请求/秒，但只能处理 100 请求/秒，
       因此过载了 20 请求/秒"
```

逐步方法发现了直接回答遗漏的过载情况。

**智能体使用 CoT 的场景：**
- 复杂工具选择决策（"给定这 5 个工具，哪一个能帮到这里？"）
- 多步骤规划（"什么行动序列能实现这个目标？"）
- 错误诊断（"工具返回了错误——哪里出了问题，下一步该怎么做？"）

### 思维树（Tree-of-Thoughts，ToT）

**是什么：** 思维链的扩展，模型探索多条推理路径，评估它们，然后选择最佳路径。

**工作原理：** 与其只有一条推理链，模型生成几种可能的方法，评分或批评每种，然后沿最有希望的路径前进。

```
目标："优化这个慢查询数据库"

路径 A："在 WHERE 子句列上添加索引"
  -> 评估："可能有效，风险低，易于实现"

路径 B："改写为物化视图"
  -> 评估："可能有帮助，但增加了复杂性和维护成本"

路径 C："对表结构进行反规范化"
  -> 评估："可能有效，但风险高，影响其他查询"

决策：先尝试路径 A，如果 A 不够就尝试路径 B
```

**智能体使用 ToT 的场景：**
- 存在多种有效方法且需要权衡取舍时
- 根本原因不确定的复杂调试
- 需要评估替代方案的架构决策

**权衡：** ToT 使用更多 token，耗时更长。将其保留给选错代价高的决策。

### 逐步分解

**是什么：** 在执行任何步骤之前，将复杂目标分解成一系列更简单的子任务。

**工作原理：** 智能体首先创建计划，然后逐步执行每个步骤，边执行边检查进度。

```
用户目标："为我们的新 API 端点设置监控"

计划：
1. 检查当前配置了哪些监控工具
2. 确定这个端点的重要指标（延迟、错误率、吞吐量）
3. 创建监控仪表板
4. 设置告警阈值
5. 测试告警是否正常触发
6. 记录监控配置

执行：[逐步进行，每个步骤可能使用工具]
```

**智能体使用分解的场景：**
- 顺序很重要的多步骤任务
- 希望智能体对其方法透明的场合
- 受益于检查点的复杂工作流程

## 模型选择：为任务选对模型

并非所有任务都需要最强大的模型。选择合适的模型是平衡能力、成本、速度和可靠性的工程决策。

### 模型谱系

```
更轻/更快/更便宜                      更重/更智能/更贵
|----------------------------------------------------------|
Gemini Flash          Gemini Pro          Gemini 2.5 Pro
（简单任务）          （均衡）            （复杂推理）
```

### 如何选择

| 任务类型 | 推荐层级 | 原因 |
|---|---|---|
| 分类（"这是垃圾邮件吗？"） | 轻量（Flash） | 简单决策，不需要复杂推理 |
| 数据提取（"从这封邮件中提取日期"） | 轻量（Flash） | 模式匹配，有明确输出 |
| 摘要 | 轻量到中等 | 取决于来源的长度和复杂性 |
| 多步骤推理 | 中等到重量（Pro） | 需要持续的逻辑链 |
| 复杂代码生成 | 重量（2.5 Pro） | 需要深入理解模式和边缘情况 |
| 智能体工具使用 | 中等到重量 | 工具选择和结果解读需要强推理能力 |
| 创意写作 | 中等 | 无需最重的模型也有好效果 |

### 模型路由：为不同步骤使用不同模型

成熟的智能体系统不会对每个步骤都使用相同的模型。这称为**模型路由**——根据复杂度将工作流的不同部分引导到不同模型。

**示例架构：**

```
用户查询到来
    |
    v
[轻量模型：意图分类]  --> "order_status"
    |
    v
[轻量模型：参数提取]  --> order_id: 12345
    |
    v
[工具调用：查询订单]  --> 状态数据
    |
    v
[轻量模型：格式化回复]  --> "您的 #12345 订单已于 3 月 15 日发货"
```

在这个流程中，每个步骤都使用快速、廉价的模型，因为没有单个步骤需要深度推理。总成本和延迟远低于整个交互都使用前沿模型的方案。

### Google Cloud 模型选项

Google Cloud 通过 Vertex AI 提供对 Gemini 模型的访问：

- **Gemini 2.5 Pro** —— 用于复杂任务的顶级推理能力。用于需要深度分析、复杂多步骤逻辑或细致理解的任务。

## 系统指令的作用

系统指令是智能体的"职位描述"。它们告诉模型它是什么、能做什么、应该如何表现，以及不应该做什么。

### 系统指令中应包含的内容

一个精心编写的智能体系统指令通常包括：

1. **角色定义**：智能体是什么，服务于谁
2. **能力**：可用的工具以及何时使用
3. **约束**：智能体不应该做的事情
4. **输出格式**：如何组织回复
5. **错误处理**：出错时该怎么做
6. **个性/语气**：如何沟通（如有必要）

### 示例：客服智能体

```
你是 Acme Corp 的客服智能体，Acme Corp 是一家在线电子零售商。

你的职责：
- 帮助客户处理订单查询、退货和产品问题
- 态度友好、专业、简洁

可用工具：
- lookup_order(order_id)：返回订单状态、商品和配送信息
- initiate_return(order_id, reason)：启动退货流程
- search_products(query)：搜索产品目录
- escalate_to_human(reason)：转接人工客服

准则：
- 在访问订单信息之前，始终验证客户身份
- 如果 3 次尝试后仍无法解决问题，将转接人工客服
- 不要虚构订单信息——始终使用查询工具
- 不要提供超出标准政策的折扣或退款
- 回复不超过 3 段

遇到工具错误时：
- 如果是临时错误（超时、500），重试一次
- 如果是永久错误（未找到、未授权），向客户解释情况
- 如果不确定，转接人工客服
```

### 编写有效系统指令的技巧

| 应该做 | 不应该做 |
|---|---|
| 明确说明工具使用方式 | 让工具选择模糊不清 |
| 定义清晰的边界 | 假设模型了解你的业务规则 |
| 包含错误处理指导 | 期待模型自行解决错误 |
| 指定输出格式 | 每次让模型自行选择格式 |
| 使用具体示例 | 写抽象、模糊的指令 |
| 保持指令简洁 | 写一篇 10 页的论文（浪费上下文） |

### 顺序很重要

模型更多关注系统提示开头和结尾的指令。请按以下方式组织系统指令：

```
1. 最关键的规则（身份、安全约束）     <-- 开头
2. 工具使用指南
3. 输出格式
4. 示例
5. 边缘情况处理
6. 最关键规则的提醒                   <-- 结尾
```

这利用了注意力中的"首因和近因"效应。

## 温度、采样与智能体行为

当语言模型生成文本时，它不会只选择单个最可能的下一个 token，而是从所有可能 token 的概率分布中采样。控制这种采样的参数对智能体行为有重大影响。

### 温度（Temperature）

**温度**控制模型输出的随机程度。

- **温度 0（或很低）**：模型几乎总是选择最可能的 token。输出是确定性的、聚焦的。
- **温度 1**：模型按 token 概率比例采样。输出更多变、更有创意。
- **温度 > 1**：模型变得越来越随机。输出变得不可预测。

**视觉类比：**

```
温度 0：    "法国的首都是巴黎。"
            （永远相同的答案）

温度 0.7：  "法国的首都是巴黎，这座城市以..."
            （阐述方式略有变化）

温度 1.5：  "法国的首都历史上根植于..."
            （更有创意，可能偏离主题）
```

### 智能体任务应使用什么温度

| 智能体任务 | 推荐温度 | 原因 |
|---|---|---|
| 工具选择 | 0 - 0.2 | 需要确定性、正确的工具调用 |
| 数据提取 | 0 | 需要精确答案，不需要创意 |
| 代码生成 | 0 - 0.3 | 正确性比多样性更重要 |
| 规划 | 0.2 - 0.5 | 一定的灵活性有助于探索选项 |
| 创意写作 | 0.7 - 1.0 | 多样性和原创性受重视 |
| 头脑风暴 | 0.8 - 1.0 | 需要多样化的想法 |

**对于大多数智能体用例，保持低温度（0 到 0.3）。** 智能体需要在工具使用、参数提取和推理方面做出可靠决策。高温度在你需要一致性的地方引入了随机性。

### Top-K 和 Top-P 采样

这些是控制模型如何选择 token 的额外参数。

**Top-K：** 只考虑概率最高的 K 个 token。如果 K=50，模型忽略前 50 之外的所有 token。

**Top-P（核采样）：** 只考虑累积概率达到 P 的 token。如果 P=0.9，模型考虑累积概率达到 90% 的最小 token 集合。

**对于智能体：** 使用保守的设置。Top-P 约 0.9 和适中的 Top-K 值是合理的默认值。模型的默认设置通常对智能体工作来说足够好——温度是你最可能想要调整的参数。

## ELI5：LLM 大脑如何工作

### 把 LLM 想象成一位厨师

想象你在经营一个餐厅厨房，LLM 就是你的主厨。

**厨师的训练（模型训练）：**
厨师花了多年研究数千本食谱书，观看烹饪节目，练习菜谱。他们没有逐字背诵每一道菜谱，但已经建立了深刻的直觉——哪些口味相配，什么技法适合什么食材，以及在缺少某样东西时如何即兴发挥。

**Token 就像食材：**
厨师不会一次性思考整道菜。他们按食材和步骤逐一思考。"先洋葱，再大蒜，再番茄……"每种食材的选择都影响下一种。这就是 token 预测的工作方式——每个 token 都基于之前所有 token 来选择。

**上下文窗口就像厨台空间：**
厨师只能用放在厨台上的东西。如果厨台很大（100 万 token），他们可以同时摆放很多食材、菜谱和准备工作。如果厨台小，他们就必须腾出空间，可能忘记自己在做什么。

**温度就像厨师的心情：**
- 低温度：厨师专注而有条理，严格按照菜谱。每次你点同一道菜，味道都一样。
- 高温度：厨师感觉有创意，即兴发挥，替换食材，尝试新东西。有时结果令人惊艳，有时则有些奇怪。

**系统指令就像餐厅的定位：**
"你是一家法式小酒馆，使用传统技法，不提供寿司。"这塑造了厨师做每个决定的方式，而无需为每道菜重复说明。

**工具就像厨房设备：**
厨师的知识本身不能烹饪食物，他需要烤箱、炉灶、刀具和量具。同样，LLM 的推理本身不能查询数据或调用 API，它需要工具。

**智能体循环就像一个烹饪节目挑战：**
厨师接到挑战（"为一位无麸质人士做一顿三道菜的晚餐"），规划方案，开始烹饪，边做边品尝，调整调味，摆盘，评估结果。如果酱汁分离了，他排障并调整。这个计划-行动-观察-调整的循环正是智能体所做的事情。

## 综合运用：模型选择如何影响智能体质量

以下是这些概念如何在真实智能体场景中结合的实际示例。

### 场景：Bug 分类智能体

你的团队想要一个智能体，它能读取新的 Bug 报告，按严重程度分类，分配给正确的团队，并起草初步调查计划。

**模型选择决策：**

| 步骤 | 模型选择 | 原因 |
|---|---|---|
| 分类严重程度（P0-P3） | Flash（轻量） | 简单分类，有清晰标准 |
| 分配给团队 | Flash（轻量） | 基于组件的查找式决策 |
| 起草调查计划 | Pro（重量） | 需要理解 Bug、相关系统，并提出诊断步骤 |

**温度决策：**

| 步骤 | 温度 | 原因 |
|---|---|---|
| 分类严重程度 | 0 | 必须确定性——同一个 Bug 应该总得到相同的严重程度 |
| 分配给团队 | 0 | 必须一致——同一个组件应该总路由到同一个团队 |
| 起草调查计划 | 0.3 | 轻微灵活性有助于生成更有用的多样化调查思路 |

**系统指令片段：**

```
你是平台工程团队的 Bug 分类智能体。

严重程度分类：
- P0：服务宕机或正在发生数据丢失
- P1：主要功能损坏，无解决方法
- P2：功能受损，但存在解决方法
- P3：小问题、外观问题或改进请求

团队路由：
- 认证/登录问题 -> 身份团队
- API 错误 -> 平台团队
- UI 问题 -> 前端团队
- 数据库/性能 -> 基础设施团队

起草调查计划时：
- 从最可能的根本原因开始
- 按优先级列出 3-5 个诊断步骤
- 包含相关日志查询或仪表板链接
- 注意可能相关的最近部署
```

这个示例展示了如何通过理解模型能力、设置适当参数和编写清晰指令，共同构建一个可靠的智能体。

## 在智能体中使用 LLM 时的常见错误

### 1. 信任模型进行数学运算

LLM 预测 token，不进行方程计算。对于任何重要的计算，使用代码执行工具。

```
错误：  "计算 47 件商品，每件 23.99 美元的总成本"
        -> 模型可能说 1127.53 美元（凑巧答对了，但经常出错）

正确：  让智能体调用计算器工具或代码执行工具
        -> calculate("47 * 23.99") -> 1127.53 美元（保证正确）
```

### 2. 假设有完美记忆

模型只"记得"当前上下文窗口中的内容。如果第 1 步的信息在第 10 步被截断，模型不会记住它。

### 3. 过度依赖单一模型

对每个步骤都使用前沿模型会浪费金钱并增加延迟。使用模型路由，将模型能力与任务复杂度匹配。

### 4. 忽视系统提示

精心设计的系统提示可能是智能体从 50% 的时间有效提升到 95% 的关键。花时间编写和迭代你的系统指令。

### 5. 不考虑幻觉

LLM 会自信地生成听起来合理但不正确的信息。对于任何重要的事实，让智能体的回答以工具结果为基础，而不是模型的训练数据。

## 核心要点

1. **LLM 是智能体的推理引擎。** 理解其能力和局限是构建好智能体的基础。

2. **Token、上下文窗口和注意力机制是三个关键概念。** Token 决定成本和速度；上下文窗口决定模型能处理多少信息；注意力机制决定模型关注什么。

3. **推理策略很重要。** 思维链、思维树和逐步分解可以大幅提升智能体在复杂任务上的表现。

4. **为每个步骤选择合适的模型。** 简单任务用轻量模型，复杂推理用重量模型。模型路由降低成本和延迟。

5. **系统指令是控制智能体行为的主要手段。** 认真编写，要具体，并根据测试结果迭代。

6. **为智能体保持低温度。** 确定性行为对生产智能体系统几乎总是更好的选择。
