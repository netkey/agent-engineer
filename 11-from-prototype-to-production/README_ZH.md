# 第十一课：从原型到生产——智能体的生产化之旅

## 简介

在餐厅开业前，厨师会在厨房里反复试验菜谱。原型阶段的菜味道不错，不代表正式营业后每道菜都能保持同样水准。

智能体同理。在你本地跑通的 demo，面对真实用户、真实流量、真实边缘案例时，表现可能完全不同。

本课介绍从原型到生产需要跨越的鸿沟：评估门控部署、安全发布策略、可观测性、成本管理和生产就绪检查清单。

---

## ELI5：餐厅开业

**原型阶段：** 你在家请朋友试菜。环境可控，客人友好，出了问题随时调整。

**生产阶段：** 餐厅正式开业。满座、翻台、食材供应链、员工排班、食品安全检查、客户投诉——一切同时发生。

从"能做出好吃的菜"到"能稳定运营一家餐厅"，中间需要：标准化流程、品控体系、应急预案、成本监控和持续改进。智能体的生产化过程同理。

---

## 原型与生产的差距

| 维度 | 原型（Demo） | 生产 |
|------|-------------|------|
| **用户** | 你（或几个人） | 成千上万 |
| **输入** | 你已知的几种问法 | 你能想到和想不到的所有问法 |
| **运行时间** | 跑几分钟就停 | 7×24 不间断 |
| **延迟要求** | "等一下就好" | P99 < 5 秒 |
| **成本考量** | 不太在意 | 每次调用都要算钱 |
| **质量门槛** | "看起来能跑" | 99%+ 准确率 |
| **安全要求** | 没有护栏也能跑 | 必须过安全审查 |
| **调试** | 本地打印日志 | 分布式追踪 + 监控告警 |
| **更新** | 改代码重新跑 | CI/CD 流水线 + 灰度发布 |

**为什么 demo 会骗人：**

- 原型只测试"快乐路径"，生产会遇到所有边缘案例
- 原型在隔离环境运行，生产有并发和资源竞争
- 原型用精心挑选的输入，生产输入是随机的
- 原型不考虑成本，生产成本可能失控

```
代码变更
  │
  ▼
┌─────────────────────────────┐
│  预合并检查（< 5 分钟）      │
│  → 代码 lint                 │
│  → 单元测试                  │
│  → 基础评估（黄金测试子集）   │
│  → 安全检查                  │
└─────────────┬───────────────┘
              │ 通过
              ▼
          合并到主分支
              │
              ▼
┌─────────────────────────────┐
│  合并后检查（< 30 分钟）     │
│  → 部署到 staging            │
│  → 完整评估套件               │
│  → 集成测试                  │
│  → 性能基准                  │
└─────────────┬───────────────┘
              │ 通过
              ▼
┌─────────────────────────────┐
│  生产门控                    │
│  → 全面评估结果审查           │
│  → 安全审查签字               │
│  → 人工批准                   │
└─────────────┬───────────────┘
              │ 批准
              ▼
          部署到生产
```

每次代码、提示或工具变更都必须重新跑评估。质量不应该随变更下降。

## 提示版本管理

提示词是代码的一部分，应该和代码一起版本管理。

```
prompts/
├── customer_service/
│   ├── v1_system_prompt.md      # 初始版本
│   ├── v2_system_prompt.md      # 添加了退款政策
│   └── current -> v2            # 当前活跃版本
├── research_agent/
│   └── v1_system_prompt.md
└── code_reviewer/
    └── v1_system_prompt.md
```

CI/CD 流程应该：
- 检测提示词变更
- 自动跑评估对比新旧提示
- 记录提示变更对质量的影响

## 安全发布策略

### 金丝雀发布（推荐）

逐步将流量从旧版本切换到新版本。

```
版本 N（当前）  →  版本 N+1（新版本）

阶段 1：5% 流量 → N+1     监控指标
阶段 2：25% 流量 → N+1    如果指标正常
阶段 3：50% 流量 → N+1    如果指标正常
阶段 4：100% 流量 → N+1   完全切换
```

如果任一阶段指标异常（完成率下降、延迟飙升、错误率上升），自动回滚。

### 蓝绿部署

两套环境并行，新版本部署到空闲环境，测试通过后切换路由。

```
蓝色（生产）── 当前服务中
绿色（staging）── 新版本部署

测试通过 → 路由切换到绿色
出现问题 → 立即切回蓝色
```

### A/B 测试

同时运行两个版本，比较哪个更好。

```
50% 流量 → 版本 A（当前提示）
50% 流量 → 版本 B（新提示）

比较：任务完成率、用户满意度、成本
```

### 功能开关

用功能开关控制智能体特性，无需重新部署即可开启/关闭。

```python
if feature_flag.enabled("agentic_rag"):
    # 使用智能体 RAG 流程
    result = agent_rag.run(query)
else:
    # 使用基础 RAG
    result = basic_rag.run(query)
```

## 成本管理

### 模型路由降低成本

| 步骤 | 模型 | 原因 |
|------|------|------|
| 意图分类 | Flash Lite | 简单分类，不需要强推理 |
| 参数提取 | Flash Lite | 模式匹配，有明确输出 |
| 核心推理 | Pro | 需要深度理解和分析 |
| 回复格式化 | Flash Lite | 简单文本生成 |

### 缓存策略

| 缓存类型 | 命中率 | 节省 |
|----------|--------|------|
| **精确匹配缓存** | 10-20% | 完全跳过 LLM 调用 |
| **语义缓存** | 5-15% | 复用相似问题的答案 |
| **工具结果缓存** | 20-40% | 跳过重复工具调用 |
| **提示缓存** | 40-80% | 利用上下文缓存减少 token |

### Token 预算管理

```python
class TokenBudget:
    def __init__(self, max_tokens: int, warning_threshold: float = 0.8):
        self.max_tokens = max_tokens
        self.used_tokens = 0
        self.warning_threshold = warning_threshold
    
    def record_usage(self, tokens: int):
        self.used_tokens += tokens
        if self.used_tokens > self.max_tokens * self.warning_threshold:
            logger.warning(f"Token usage at {self.used_tokens/self.max_tokens:.0%}")
        if self.used_tokens > self.max_tokens:
            raise TokenBudgetExceeded(f"Used {self.used_tokens}, limit {self.max_tokens}")
```

### 成本监控分级

| 级别 | 指标 | 频率 |
|------|------|------|
| **每次请求** | token 消耗、成本 | 实时 |
| **每小时** | 总成本、平均成本/请求 | 每小时聚合 |
| **每日** | 日总成本、趋势、异常 | 每日报告 |
| **每月** | 月度预算、按智能体/用户拆分 | 月度审查 |

## 核心要点

1. **原型到生产是质的飞跃。** demo 能跑不代表能服务真实用户。

2. **评估门控是质量保障。** 每次变更都必须通过评估才能上线。

3. **渐进式发布降低风险。** 金丝雀发布（5% → 25% → 50% → 100%）在问题扩大前捕获。

4. **可观测性不是可选项。** 没有日志、追踪和指标，生产问题无法调试。

5. **成本需要主动管理。** 模型路由、缓存策略和 token 预算缺一不可。

6. **提示是代码。** 版本管理、评估对比、和代码一样的流程。

7. **持续迭代是常态。** 观察→行动→演进循环永不停止。
