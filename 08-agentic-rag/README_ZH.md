# 第八课：智能体 RAG——超越基础检索

## 简介

基础 RAG（检索增强生成）就像开卷考试：学生拿到问题后翻书找答案，找到什么就用什么。但如果翻到的那页恰好不包含正确答案呢？学生会给出错误答案。

智能体 RAG 更像一个研究助理——不只是翻书，还会判断找到的信息是否相关、是否需要换方式搜索、是否要查阅多个来源交叉验证，最终给出可靠的答案。

本课介绍智能体 RAG 与基础 RAG 的区别、核心能力、工作流程和实现模式。

---

## ELI5：图书管理员与研究助理

**基础 RAG 像图书管理员。** 你给一个问题，她快速检索系统中最匹配的文档，返回给你。检索是机械的——基于关键词或向量相似度，不理解上下文。

**智能体 RAG 像研究助理。** 你给一个问题，他会：
- 理解你真正需要什么
- 选择合适的搜索方式
- 评估找到的信息是否可靠
- 如果信息不足，换个角度再搜
- 综合多个来源给出有根据的答案

区别：一个是检索，一个是研究。

## 基础 RAG 的问题

基础 RAG（检索→生成）在简单场景下有效，但面对复杂查询时暴露诸多问题：

### 1. 查询-文档不匹配

用户问："我们的 Q3 营收相比 Q2 增长了多少？"

RAG 检索：返回包含"Q3"和"营收"的文档，但没有具体的对比数据。

结果：模型基于不完整信息编造答案。

### 2. 多文档回答

用户问："对比 AWS 和 GCP 的服务器定价。"

基础 RAG：可能只检索到一个云服务商的文档，缺少对比所需的双方信息。

### 3. 首个结果不足

如果第一次检索没有找到相关内容，基础 RAG 就此止步——不会重试或调整查询。

### 4. 需要分解的查询

用户问："使用我们推荐的技术栈创建一个 Web 应用，部署到生产环境。"

这需要分步骤检索（框架选择→数据库→部署配置），基础 RAG 无法分解。

### 5. 缺乏验证

基础 RAG 不验证检索到的信息是否正确、是否过时、是否与查询相关。

---

## 智能体 RAG 的能力

智能体 RAG 通过以下方式解决上述问题：

### 1. 自主查询规划

智能体不只是用用户原始查询检索——它决定需要什么信息、如何分解问题、使用哪些检索策略。

```
用户："对比 AWS 和 GCP 的服务器定价"

智能体规划：
  1. 搜索 AWS 服务器定价
  2. 搜索 GCP 服务器定价
  3. 对比两者
  4. 生成对比表格
```

### 2. 自适应源选择

智能体根据查询类型选择合适的数据源。

```
技术问题 → 查阅技术文档
财务问题 → 查阅财报/财务数据
人员问题 → 查阅内部目录/组织架构
```

### 3. 上下文感知扩展

如果初步检索的信息不够，智能体会自动扩展搜索范围。

```
第一次搜索："Q3 营收增长" → 数据不完整
扩展搜索："Q2 营收" + "Q3 财报" + "季度对比" → 足够数据
```

### 4. 多跳推理

智能体能执行"检索→推理→再检索"的多跳循环。

```
跳 1：查找 "公司产品 X 的技术负责人" → 找到张工
跳 2：查找 "张工负责的另一个项目" → 找到项目 Y
跳 3：查找 "项目 Y 的当前状态" → 返回结果
```

基础 RAG 只能做一跳（直接检索），智能体 RAG 能串联多跳。

### 5. 自我修正

智能体评估检索结果，如果质量不够则调整策略。

```
智能体："检索到的 3 篇文档都不包含具体定价数据。
        需要换搜索关键词，或直接查阅定价页面。"
→ 重新搜索："2024 AWS EC2 按需定价表"
```

## 智能体 RAG 工作流程

```
第 1 步：查询规划
  分析用户问题，决定需要什么信息、如何检索。

第 2 步：检索
  从合适的源获取信息。可能涉及多个独立查询。

第 3 步：评估结果
  检查结果是否相关、完整、可靠。
  "这些文档回答了问题吗？"

第 4 步：精炼（如果需要）
  如果结果不足，调整查询策略并重新检索。
  "信息不完整——需要查找其他来源。"

第 5 步：生成答案
  基于检索到的信息生成回答。

第 6 步：验证
  验证答案是否基于检索结果而非幻觉。
  "我的回答有文档支持吗？"
```

这个循环可以在步骤 3-4 之间迭代多次，直到信息充足或达到最大迭代次数。

---

## 基础 RAG vs. 智能体 RAG

| 方面 | 基础 RAG | 智能体 RAG |
|------|----------|-----------|
| **查询** | 直接使用用户查询 | 可重构、分解、扩展查询 |
| **检索** | 单次检索 | 多次迭代检索 |
| **评估** | 不评估结果质量 | 评估相关性并自我修正 |
| **源** | 固定单一源 | 自适应选择数据源 |
| **推理** | 单跳 | 多跳（检索→推理→再检索） |
| **验证** | 不验证 | 验证答案有文档支持 |
| **成本** | 低（1-2 次 LLM 调用） | 高（多次迭代） |
| **延迟** | 低 | 较高 |
| **准确性** | 中等 | 高 |

## 综合运用：技术支持智能体

**场景：** 用户报告产品问题，智能体需要从知识库中查找解决方案。

```
用户："部署到 Kubernetes 时，pod 一直 CrashLoopBackOff。"

智能体 RAG 工作流：

1. 查询规划：
   "需要查找：CrashLoopBackOff 原因、Kubernetes 部署故障排查、
               我们产品的已知部署问题。"

2. 第一次检索：
   搜索知识库 "CrashLoopBackOff Kubernetes"
   → 返回 3 篇通用 K8s 排障文章

3. 评估：
   "这些文章讲通用 K8s 问题，没有针对我们产品的内容。
    需要更具体的搜索。"

4. 精炼搜索：
   搜索 "CrashLoopBackOff 产品名 部署"
   → 找到知识文章 #42："部署时内存限制过低导致 OOM"

5. 多跳推理：
   "文章 #42 提到内存限制——检查用户配置中的资源限制。
    搜索知识库 '资源限制 默认值'"
   → 找到默认资源配置文档

6. 生成答案：
   "CrashLoopBackOff 很可能因为内存限制过低导致 OOM。
    建议将 memory limit 从 256Mi 提升到 512Mi。
    具体配置参考知识文章 #42..."

7. 验证：
   答案有知识文章 #42 和配置文档支持。验证通过。
```

## 实现模式

### 1. 查询分解

将复杂查询拆分为多个子查询，分别检索后综合。

```
原始查询："对比我们产品中三个定价方案的功能差异"
子查询：
  - "基础版包含哪些功能"
  - "专业版包含哪些功能"
  - "企业版包含哪些功能"
综合：生成对比表格
```

### 2. 带验证的检索

检索后验证结果质量，不足则重新检索。

```
检索 → 评估（相关度 < 阈值？） → 是 → 调整查询重新检索
                              → 否 → 使用结果生成答案
```

### 3. 迭代深入

从宽泛搜索开始，逐步聚焦到具体内容。

```
第一次：搜索 "Kubernetes 部署问题" → 范围太广
第二次：搜索 "Kubernetes CrashLoopBackOff 产品名" → 更相关
第三次：搜索 "产品名 内存限制 OOM" → 找到具体原因
```

### 4. 来源三角验证

从多个独立来源检索，交叉验证信息一致性。

```
来源 A：内部文档 → "默认内存限制 256Mi"
来源 B：配置文件 → "resources.limits.memory: 256Mi"
来源 C：工单系统 → "3 个类似工单，都是内存不足"
→ 三个来源一致，结论可信
```

- 根据查询选择数据源
- 评估检索结果并迭代

## 常见陷阱

### 1. 无限检索循环

智能体反复搜索同一个东西而没有进展。

**修复：** 设置最大检索迭代次数（通常 3-5 次）。

### 2. 过度检索

检索大量不相关信息，淹没上下文窗口。

**修复：** 限制每次检索的结果数量，只保留高相关度文档。

### 3. 忽视上下文

智能体完全依赖检索结果，忽视用户提供的上下文。

**修复：** 始终将用户输入作为最高优先级上下文。

### 4. 没有回退

检索无结果时智能体无计可施。

**修复：** 设计回退路径——"我没有找到具体答案，但基于我的知识..."或转接人工。

---

## 核心要点

1. **智能体 RAG = 检索 + 推理 + 自我修正。** 不只是查找文档，而是理解需要什么、找到了什么、还缺什么。

2. **基础 RAG 的问题是机械检索。** 查询不匹配、单跳限制、无验证——智能体 RAG 通过循环解决这些问题。

3. **多跳推理是关键差异。** 智能体能串联"检索→推理→再检索"，基础 RAG 只能做一次检索。

4. **自我修正提升可靠性。** 智能体评估检索质量并调整策略，而不是盲目使用不充分的信息。

5. **成本更高但准确性更好。** 智能体 RAG 需要更多 LLM 调用和更长延迟，但结果显著更可靠。

6. **设置迭代限制。** 防止无限检索循环——3-5 次通常足够。

7. **设计回退路径。** 检索无结果时，智能体应该有备用的回应方式。
