# 第五课：记忆与上下文——智能体如何记忆

## 简介

如果你每次对话都从头开始——忘记上次说的名字、聊过的话题、达成的约定——你很快就会觉得对方根本不在乎。这就是没有记忆的智能体。

记忆对智能体的重要性不亚于大脑。没有记忆，智能体无法追踪已完成的工作、用户的偏好或之前的对话。每个请求都像第一次遇到。

本课介绍智能体如何管理记忆：短期（会话内）、长期（跨会话）以及管理上下文窗口的策略。

---

## ELI5：上下文窗口就像书桌

想象你在书桌上工作。桌子越大，你能同时摊开的文件、笔记和参考资料就越多。小桌子意味着你需要不断收起旧文件腾出空间——可能会忘记之前在读什么。

**上下文窗口就是智能体的书桌。** 它决定智能体一次能"记住"多少信息。桌子大（100 万 token）可以放很多文件。桌子小（8000 token）需要频繁清理。

随着智能体执行多个步骤——调用工具、获取结果、继续推理——桌面越来越满。最终没有空间放新文件。这就是**上下文耗尽**。

管理上下文就是管理书桌：
- **滑动窗口：** 收走最旧的文件，为新文件腾出空间
- **摘要：** 把一大叠笔记浓缩成几行要点
- **优先级筛选：** 只保留当前任务最相关的文件
- **外部存储：** 把重要信息写在便签上贴到桌边（长期记忆）

## 上下文工程：提示工程的演进

提示工程关注如何写好单次提示。上下文工程关注如何管理智能体在整个生命周期中访问的信息。

```
提示工程：    "写好一个问题"
上下文工程：  "管理智能体看到的所有信息"
```

上下文工程包括：
- 哪些信息放入上下文窗口
- 何时丢弃或摘要旧信息
- 如何从外部存储检索相关信息
- 平衡信息完整性与上下文限制

好的上下文工程让智能体看到它需要的信息，不多不少。

## 记忆类型

智能体有几种不同的记忆类型，每种服务于不同目的：

| 类型 | 持续范围 | 存储内容 | 类比 |
|------|----------|----------|------|
| **短期记忆** | 单个会话 | 当前对话历史、工具结果、中间状态 | 工作记忆——你在思考什么 |
| **长期记忆** | 跨会话 | 用户偏好、历史交互、持久化知识 | 长期记忆——你记住的事情 |
| **程序性记忆** | 持久化 | 如何执行任务、最佳实践、模式 | 肌肉记忆——你知道怎么做 |
| **声明性记忆** | 持久化 | 事实、数据、知识库 | 事实记忆——你知道什么 |

### 短期记忆（会话记忆）

短期记忆是智能体的工作记忆。它包含当前会话中发生的一切：用户消息、智能体回复、工具调用和结果。

**特点：**
- 存储在上下文窗口中
- 随会话结束而消失
- 受上下文窗口大小限制
- 对智能体的即时推理至关重要

**挑战：** 随着会话变长，上下文窗口可能装满。如果不管理，智能体会"忘记"早期对话内容。

### 长期记忆

长期记忆跨会话持久化。它存储用户偏好、历史交互结果、以及智能体学到的持久知识。

**特点：**
- 存储在数据库、向量存储或知识图谱中
- 跨会话持久化
- 需要有意识的写入和检索策略
- 使智能体"认识"用户

**示例：**
- "用户总是将日期格式化为 YYYY-MM-DD"
- "上次讨论时，用户决定使用 PostgreSQL 而非 MySQL"
- "项目使用 Python 3.11，依赖管理用 Poetry"

### 程序性记忆

程序性记忆是关于"如何做"的知识——流程、模式、最佳实践。

**特点：**
- 编码在系统指令、工具定义或技能文件中
- 告诉智能体"怎么做事情"
- 通常由开发者预设，也可从经验中学习

**示例：**
- "部署前先运行测试"
- "错误处理模式：重试一次，然后回退"
- "代码风格：使用 f-string 而非 .format()"

### 声明性记忆

声明性记忆是关于"是什么"的知识——事实、数据、参考信息。

**特点：**
- 存储在文档、数据库或知识库中
- 告诉智能体"事实是什么"
- 通常通过 RAG（检索增强生成）检索

**示例：**
- API 文档
- 产品目录
- 公司政策
- 技术规格

## 会话：短期记忆的容器

会话是短期记忆的组织单元。它将持续时间内的所有交互分组为一个连贯的上下文。

### 会话包含的内容

```
会话 ID：session_abc123
用户：user_456
创建时间：2024-03-15T10:00:00Z
最后活跃：2024-03-15T10:15:00Z

事件：
  [10:00:00] 用户："帮我查一下订单 #12345"
  [10:00:01] 智能体：调用 lookup_order("12345")
  [10:00:02] 工具结果：{"status": "shipped", ...}
  [10:00:03] 智能体："您的订单 #12345 已发货..."
  [10:05:00] 用户："预计什么时候到？"
  [10:05:01] 智能体：（从会话历史中找到订单信息）
             "预计 3 月 20 日送达"
```

没有会话，第二个问题"预计什么时候到？"就没有订单号的上下文。会话让智能体能够理解后续引用。

### ADK 中的会话

[Agent Development Kit（ADK）](https://google.github.io/adk-docs/) 提供内建的会话管理：

- **会话状态** —— 跨请求持久化对话历史
- **事件日志** —— 记录所有交互和工具调用
- **上下文组装** —— 自动将历史注入上下文窗口
- **会话生命周期** —— 可配置的超时和清理策略

### Vertex AI Agent Engine 中的会话

[Vertex AI Agent Engine](https://cloud.google.com/vertex-ai/generative-ai/docs/agent-engine/overview) 在托管层面管理会话：

- 自动会话存储和检索
- 跨实例的会话共享
- 与 Google Cloud 监控集成
- 会话级指标（持续时间、交互次数、错误率）

## 记忆 vs. RAG

记忆和 RAG（检索增强生成）经常混淆。它们互补但有不同目的。

| 方面 | 记忆 | RAG |
|------|------|-----|
| **类比** | 个人助手记住你的偏好 | 图书馆查找参考信息 |
| **存储** | 数据库、向量存储、键值存储 | 向量数据库、文档索引 |
| **检索** | 基于会话上下文自动注入 | 按需检索相关文档 |
| **内容** | 个人化、交互历史、状态 | 文档、文章、知识库 |
| **持久性** | 跨会话持久化 | 外部数据源，更新独立 |
| **示例** | "用户上次选择蓝色" | "2024 年云安全最佳实践" |

**简单区分：**
- 记忆 = 智能体从过去交互中"记住"的内容
- RAG = 智能体从外部知识库"查阅"的内容

大多数生产智能体同时使用两者：记忆处理个人上下文，RAG 处理领域知识。

## 上下文腐化

随着智能体积累上下文，早期信息可能变得不相关或误导。这称为**上下文腐化**。

### 腐化的原因

1. **过时信息。** 早期工具结果不再适用（"库存：有货" → 后来售罄）。
2. **矛盾信息。** 后续步骤推翻了之前的结论。
3. **信息淹没。** 关键指令或发现淹没在大量不相关文本中。
4. **推理漂移。** 智能体基于旧上下文偏离原始目标。

### 腐化的迹象

- 智能体重复已经回答的问题
- 智能体使用过时的数据做决策
- 智能体偏离原始目标
- 响应质量随会话长度下降

### 应对策略

**1. 滑动窗口**

丢弃最旧的消息，保留最近的 N 条。简单但可能丢失重要早期上下文。

**2. 摘要**

定期将历史对话压缩为摘要。保留要点的同时大幅减少 token 数。

```
原始（5000 token）：完整的 10 轮对话
摘要（500 token）："用户询问了订单 #12345 的配送状态。
                   确认已发货，预计 3 月 20 日送达。
                   用户随后询问了退货政策。"
```

**3. 基于 Token 的截断**

当上下文接近窗口限制时，截断最长的工具结果，保留关键信息。

**4. 基于重要性的选择**

给不同信息分配重要性分数。保留高分项目，丢弃低分项目。

```
高重要性：用户请求、最终答案、错误信息
中重要性：工具调用、中间推理步骤
低重要性：冗长工具输出、重复确认
```

**5. 外部化 + 检索**

将完整历史存储到外部数据库，只在需要时检索相关部分。这是记忆系统的核心模式。

## 记忆存储

长期记忆需要持久化存储。选择取决于数据类型和访问模式。

### 向量数据库

向量数据库存储嵌入（文本的数字表示），支持语义搜索——按含义而非关键词查找。

**适用场景：**
- 查找语义相关的过去交互
- 从文档库中检索相关内容
- 基于含义的相似性匹配

**Google Cloud：** [Vertex AI Vector Search](https://cloud.google.com/vertex-ai/docs/vector-search/overview) 提供托管的向量相似性搜索。

### 知识图谱

知识图谱存储实体及其关系。适合表示结构化知识。

**适用场景：**
- 实体间有复杂关系（用户→订单→产品→类别）
- 需要推理关系（"查找购买了 X 也购买了 Y 的用户"）
- 维护本体和分类法

### 混合方法

大多数生产系统混合使用多种存储：

```
向量数据库 → 语义搜索过去交互和文档
关系数据库 → 结构化数据（用户、订单、会话）
键值存储   → 快速查找（会话状态、用户偏好）
知识图谱   → 实体关系推理
```

## 综合运用：记忆管理的智能体

以下是记忆如何在真实智能体中工作的示例。

**场景：** 客服智能体，能记住用户历史和偏好。

**记忆架构：**

```
短期记忆（会话内）：
  - 当前对话历史
  - 最近查询的订单信息
  - 当前任务状态

长期记忆（跨会话）：
  - 用户资料（姓名、偏好语言、时区）
  - 历史订单和交互
  - 过去的偏好和决定

程序性记忆（系统指令）：
  - "始终先验证身份再访问订单信息"
  - "如果 3 次尝试未解决，转接人工"
  - "回复不超过 3 段"

声明性记忆（RAG）：
  - 产品目录
  - 退货政策文档
  - 配送时间表
```

**工作流：**

```
1. 用户发起会话
2. 从长期记忆加载用户资料 → "欢迎回来，张三"
3. 用户："我上次买的东西到了吗？"
4. 从短期记忆（如果有）或长期记忆检索最近订单
5. 查询物流 → "您的订单 #12345 预计明天送达"
6. 会话结束 → 将交互摘要写入长期记忆
7. 下次会话时，智能体记住这次交互
```

## 核心要点

1. **记忆使智能体有连续性。** 没有记忆，每次交互都是从零开始。

2. **短期记忆 = 会话上下文。** 它使智能体能够理解后续引用和维持连贯对话。

3. **长期记忆跨会话持久化。** 它使智能体"认识"用户并从过去交互中学习。

4. **上下文窗口是有限的书桌空间。** 管理它：摘要、截断、滑动窗口、外部化。

5. **上下文腐化是真实问题。** 过时或矛盾的信息会降低智能体质量。主动管理上下文新鲜度。

6. **记忆和 RAG 互补。** 记忆处理个人上下文，RAG 处理领域知识。大多数智能体需要两者。

7. **上下文工程是提示工程的演进。** 关注智能体访问的整体信息管理，而不只是单次提示的质量。
